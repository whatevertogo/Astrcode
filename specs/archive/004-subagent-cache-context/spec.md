# Feature Specification: 子智能体会话与缓存边界优化

**Feature Branch**: `004-subagent-cache-context`  
**Created**: 2026-04-09  
**Status**: Draft  
**Input**: User description: "将子智能体架构优化为独立子会话模型，并补齐存储分离、共享渲染缓存、结构化上下文继承、父子唤醒通知、lineage 索引与 resume 语义，使父子 Agent 的 durable 边界、缓存契约和恢复行为稳定且可验证。"

## Clarifications

### Session 2026-04-09

- Q: 缓存复用与父侧唤醒是否应视为同一用户故事？ → A: 否，分别作为 Story 4 的稳定缓存复用和 Story 5 的父侧可靠唤醒建模，避免验收测试耦合。
- Q: 缓存复用判断的强契约由哪些维度组成？ → A: 由规范化工作目录、活动配置名称、排序后的允许工具集合、规则文件内容摘要和上下文构建版本组成强指纹，任一维度变化都视为新缓存键。
- Q: 父智能体忙碌时收到多个子交付应如何处理？ → A: 各次交付必须独立缓冲并在父智能体空闲后可靠逐个送达，不得丢失、合并或重复消费。
- Q: recent tail 的语义筛选是否允许为此额外发起推理？ → A: 默认不允许，优先采用角色优先级、预算裁剪和工具输出摘要等确定性规则。
- Q: lineage 不一致导致无法安全恢复时系统如何暴露问题？ → A: 必须同时写入父侧可见的边界错误事实和系统诊断记录，并阻止本次恢复继续。
- Q: resume 是否要求把 compaction 或 recovery 材料固化进单一状态结构？ → A: 否，规格约束完整恢复语义而不绑定内部结构；resume 必须通过 child session 的 durable replay 恢复下一轮执行所需的完整可见状态，至少覆盖消息历史与 phase，compaction 与 recovery 材料可由 projector 或 context pipeline 等价重建。
- Q: 父唤醒约束应落在哪一层？ → A: durable 事件流只记录子交付等边界事实，不记录 ReactivationPrompt 一类机制性消息；父唤醒通过运行时信号触发，交付详情通过一次性结构化输入参与下一轮处理。
- Q: 结构化继承应落在哪一层？ → A: `resolve_context_snapshot()` 只产出任务主体；父背景通过 `PromptDeclaration` 注入 system blocks，不进入消息流。
- Q: 缓存 key contract 应该在规格里写死字段还是委托现有 fingerprint 体系？ → A: 规格只约束可观察行为与失效边界，具体 key 计算委托给 `runtime-prompt` 的 fingerprint 体系，确保工作目录、profile、工具、规则与 prompt 构建版本等变化都会触发失效。
- Q: SC-003 是否需要限定 provider 的缓存指标能力？ → A: 需要。`SC-003` 仅在支持缓存指标的 provider 上以 `cache_creation_input_tokens` 验收；不支持该字段的 provider 必须在本 feature 交付前补齐等价缓存可观测指标，否则暂不纳入本条验收范围。
- Q: 对旧共享写入历史是否还要保留兼容读取？ → A: 不保留。为保持干净架构，本 feature 完成后 legacy 共享写入历史不再保证可读；系统必须显式返回 unsupported 或 upgrade required，而不是继续维护双轨 durable 语义。
- Q: `ChildSessionNode` 是否应替代 `SubRun` 成为唯一子协作主语？ → A: 不应替代。保留 `SubRun` 作为一次子执行实例；`ChildSessionNode` 作为稳定 durable 子会话节点。后续读模型和 UI 可以降低对 `SubRun` 线程树的中心依赖，但不得把执行实例与子会话身份合并成单一概念。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 子智能体拥有独立会话真相 (Priority: P1)

作为架构维护者，我希望每个子智能体从创建开始就拥有独立的会话身份和独立的持久化历史，这样父子边界清晰，恢复、排障和后续演进都不需要从一份混合历史里反推真实关系。

**Why this priority**: 这是本特性的根边界。只要父子仍共享一份历史，lineage、resume、投影和缓存优化都会建立在不可靠前提上。

**Independent Test**: 在同一父会话下连续创建多个子智能体，验证每个子智能体都有独立会话身份和独立历史，且父会话历史只保留边界事件与结果摘要。

**Acceptance Scenarios**:

1. **Given** 父智能体发起一次新的子任务委派，**When** 系统创建子智能体，**Then** 系统为该子智能体创建新的子会话身份，并将其与本次子执行实例（`SubRun`）关联。
2. **Given** 同一父会话连续启动多个子智能体，**When** 这些子智能体运行并产生中间活动，**Then** 它们分别写入各自的持久化历史，而不是混写到父会话历史中。
3. **Given** 父会话查看自己的历史，**When** 某个子智能体运行、失败或完成，**Then** 父会话只看到边界事件、可消费通知和终态交付摘要，而不是子智能体的完整内部过程。
4. **Given** 系统读取旧历史数据，**When** 旧数据仍来自共享写入模式，**Then** 系统明确返回 unsupported 或 upgrade required，并拒绝继续按旧 durable 语义读取、恢复或投影，而不是继续维护共享写入兼容路径。

### User Story 2 - 恢复沿用原子会话而不是重开新会话 (Priority: P1)

作为架构维护者，我希望子智能体被恢复时继续使用原来的子会话身份，并基于已有历史恢复上下文，这样恢复后的行为、入口和关系都与原子会话保持一致。

**Why this priority**: 如果恢复被实现成“重新创建一个相似的子智能体”，那么 lineage、会话入口、观察视图和缓存收益都会失真。

**Independent Test**: 让子智能体在中途停止后再执行恢复，验证恢复前后子会话身份不变、子执行实例（`SubRun`）变化、恢复后继续基于原历史工作。

**Acceptance Scenarios**:

1. **Given** 某个子智能体已经存在并保留完整历史，**When** 系统恢复该子智能体，**Then** 系统沿用原子会话身份，并为本次恢复分配新的子执行实例（`SubRun`）标识。
2. **Given** 某个子智能体在恢复前已积累消息、摘要和活动记录，**When** 恢复开始，**Then** 系统通过该子会话 durable 历史的 replay 或等价 projector 重建下一轮执行所需的完整可见状态，至少恢复消息历史与 phase，而不是从空状态重新开始。
3. **Given** 父会话中已经存在该子智能体的稳定入口，**When** 子智能体被恢复，**Then** 父会话继续通过同一入口观察和管理它，并收到明确的已恢复通知。
4. **Given** lineage 信息缺失、冲突或历史损坏导致无法安全恢复，**When** 用户或父智能体发起恢复，**Then** 系统明确返回恢复失败，而不是悄悄创建新的并列子会话。

### User Story 3 - 父背景以结构化方式继承给子智能体 (Priority: P2)

作为架构维护者，我希望父智能体传给子智能体的背景信息是结构化继承的，而不是直接拼接到第一条任务消息里，这样子智能体能区分“我要做什么”和“我继承了什么背景”。

**Why this priority**: 任务和背景混写会降低任务清晰度，也会使缓存复用和上下文治理持续变差。

**Independent Test**: 在同一父会话下启动多个相似子智能体，验证子智能体的首条任务消息只描述任务本身，父背景通过独立继承块传递。

**Acceptance Scenarios**:

1. **Given** 父会话中存在压缩摘要与最近活动，**When** 子智能体启动，**Then** 系统通过 `PromptDeclaration` 将父背景以独立 system block 形式传递给子智能体，而不是将其拼接进首条任务消息。
2. **Given** 父背景中的摘要变化较少、最近活动变化较快，**When** 系统构建继承背景，**Then** 两类背景以独立继承块和独立缓存边界管理，避免高频变化内容拖累低频稳定内容。
3. **Given** 最近活动中包含长文本或大量工具输出，**When** 系统为子智能体准备继承背景，**Then** 系统优先保留语义关键项与摘要，而不是无界继承原文。
4. **Given** 子智能体收到继承背景与任务说明，**When** 它开始工作，**Then** 它能够基于清晰任务消息和独立 system block 中的结构化背景继续执行，而不需要从单条混合消息中自行拆分语义。

### User Story 4 - 相似子智能体安全复用稳定缓存 (Priority: P2)

作为架构维护者，我希望父子智能体在安全条件下共享稳定上下文缓存，这样重复启动相似子智能体时能够减少重复准备成本，同时不牺牲 prompt 正确性。

**Why this priority**: 当前重复渲染直接放大了多子智能体场景下的 token 浪费；如果缓存契约不明确，又会引入比性能问题更难排查的错误命中。

**Independent Test**: 在相同工作条件下重复启动相似子智能体，验证后续启动的稳定上下文准备成本显著下降，且在关键上下文维度变化时不会误命中旧缓存。

**Acceptance Scenarios**:

1. **Given** 父子智能体运行在相同工作条件下，**When** 子智能体启动，**Then** 系统复用可安全共享的稳定上下文准备结果，减少重复准备开销。
2. **Given** 工作目录、活动配置、允许工具集合、规则内容、prompt declarations、skills 或上下文构建版本发生变化，**When** 系统判断是否复用缓存，**Then** 系统只在 `runtime-prompt` fingerprint 体系判断强指纹完全一致时复用，否则重新准备上下文。
3. **Given** 两次上下文内容不同但规模相似，**When** 系统计算缓存复用判断，**Then** 系统不会因为弱特征相似而错误命中旧缓存。

---

### User Story 5 - 子交付可靠唤醒父智能体且不污染历史 (Priority: P2)

作为架构维护者，我希望子智能体交付结果后能可靠唤醒父智能体继续工作，但不会把变化的交付详情写进父消息历史，从而破坏父侧缓存连续性。

**Why this priority**: 交付是父子协作闭环的最后一步；如果这一层处理不稳定，多子智能体协作会出现丢交付、重复消费或父缓存持续失效。

**Independent Test**: 让多个子智能体在父 turn 结束后依次交付结果，验证父智能体都能被运行时唤醒继续处理，且父 durable 历史只保留子交付边界事实，不出现携带交付详情的可变任务消息或机制性唤醒消息。

**Acceptance Scenarios**:

1. **Given** 子智能体在父 turn 结束后完成工作，**When** 系统将子结果交付给父智能体，**Then** 父智能体通过运行时唤醒信号被可靠触发继续处理该结果，而不是依赖向 durable 历史追加机制性消息。
2. **Given** 子智能体交付包含摘要、状态和查看入口，**When** 父智能体构建下一轮上下文，**Then** 这些交付详情以一次性结构化输入参与本轮处理，并在消费后清除；父 durable 历史只保留子交付边界事实，而不是写成可变任务消息。
3. **Given** 父智能体正在处理其他工作时又收到新的子交付，**When** 当前工作尚未结束，**Then** 系统可靠保留这些交付，并在父智能体可继续处理时逐个送达，而不丢失也不错误合并。
4. **Given** 子智能体被父主动取消或被强制终止，**When** 其进入终态，**Then** 父侧仍收到明确的终态边界事实和可读摘要，而不是留下无结论的悬挂状态。

---

### Edge Cases

- 同一父会话短时间内并发创建多个子智能体时，父历史不得混入任何一个子智能体的完整内部过程。
- 子智能体恢复时，如果父侧记录与子侧元数据不一致，系统必须显式暴露 lineage 不一致，而不是猜测父子关系。
- 两次上下文内容不同但表面规模相似时，系统不得错误复用缓存。
- 最近活动中包含超长工具输出时，系统必须提炼为有用摘要，而不是简单取消截断后直接继承原文。
- 父背景的结构化继承必须通过 prompt 层注入，不得把 compact summary 或 recent tail 回写成子消息流中的普通 `UserMessage`。
- 子智能体在父 turn 结束后交付结果时，如果父智能体当前空闲，应被可靠唤醒；如果父智能体正在处理其他工作，也不得丢失该交付。
- 运行时实时观测信息在进程重启后可以消失，但 durable 真相、父子关系和子会话入口必须保留。
- 如果一次性交付输入在消费前因进程重启而丢失，系统仍必须通过持久化边界事实让维护者追溯交付发生过。
- 父侧 durable 历史不得为了触发唤醒而写入 `ReactivationPrompt` 一类机制性 `UserMessage`；唤醒机制与交付事实必须分层表达。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 为每个新创建的子智能体分配新的子会话身份，并将该身份与父会话和本次子执行实例（`SubRun`）建立明确关联。
- **FR-002**: 系统 MUST 区分“子会话身份（由 `ChildSessionNode` 表达）”和“子执行实例身份（`SubRun`）”，允许同一子会话拥有多次子执行实例。
- **FR-003**: 系统 MUST 将子智能体的消息、活动、摘要、终态和恢复所需信息持久化到子会话自己的历史中。
- **FR-004**: 父会话历史 MUST 只记录子任务的边界事件、可消费通知和终态交付摘要。
- **FR-005**: 父会话历史 MUST NOT 直接承载子智能体的完整内部过程。
- **FR-006**: 系统 MUST 区分父对子智能体的持久化认知与运行时实时观测；运行时实时观测可以依赖内存态和广播，但不得成为唯一 durable 真相。
- **FR-007**: 系统 MUST 为父会话、`ChildSessionNode`、子会话和子执行实例之间维护可查询的 lineage 关系。
- **FR-008**: 当父侧边界事实与子侧元数据冲突时，系统 MUST 显式标记关系不一致，而不是伪造 ancestry。
- **FR-009**: 系统 MUST 在恢复子智能体时沿用原子会话身份，而不是创建新的并列子会话。
- **FR-010**: 系统 MUST 通过子会话 durable 历史的 replay 或等价 projector 重建恢复下一轮执行所需的完整可见状态，至少覆盖消息历史与 phase；compaction 与 recovery 相关材料不要求固化到单一状态结构，但 MUST 能基于 durable 历史经 projector 或 context pipeline 等价重建，而不是以空状态重建。
- **FR-011**: 系统 MUST 在恢复成功后向父侧产生明确的已恢复通知。
- **FR-012**: 当无法安全恢复时，系统 MUST 返回明确失败，不得静默降级为重新 spawn。
- **FR-013**: 所有新创建的子智能体 MUST 默认采用独立子会话模式；系统 MUST 删除旧共享写入模式的读取、回放与恢复兼容路径，并在遇到 legacy 共享写入历史时明确返回 unsupported 或 upgrade required。
- **FR-014**: 系统 MUST 支持在安全条件下复用父子之间的稳定上下文准备结果，以减少重复启动相似子智能体的准备开销。
- **FR-015**: 缓存复用判断 MUST 能区分工作目录、活动配置、允许工具集合、规则内容、prompt declarations、skills 与上下文构建版本等可观察输入的变化。
- **FR-016**: 系统 MUST NOT 仅依据长度、数量或其他弱特征判断缓存是否可以安全复用。
- **FR-017**: 父传子的背景信息 MUST 与子任务消息分离表达；`resolve_context_snapshot()` 产出的任务主体只描述任务目标与直接上下文，不承载父摘要或最近活动全文。
- **FR-018**: 系统 MUST 将父的压缩摘要与最近活动作为独立继承部分，通过 `PromptDeclaration` 注入 system blocks 传给子智能体，以便它们独立更新和复用，而不经过消息流。
- **FR-019**: 系统 MUST 对最近活动执行语义筛选与摘要化，优先保留关键用户输入、关键结论和必要的活动摘要。
- **FR-020**: 系统 MUST 避免将大型工具输出或低价值噪音无界继承给子智能体。
- **FR-021**: 子智能体的首条任务消息 MUST 能单独表达任务目标，而不包含父背景全文拼接；父背景必须通过独立的 prompt system blocks 参与构建。
- **FR-022**: 当子智能体进入终态时，系统 MUST 向父侧生成结构化交付，至少包含子会话身份、子执行实例（`SubRun`）、终态状态、可读摘要和查看入口。
- **FR-023**: 系统 MUST 支持在父 turn 已结束后通过运行时唤醒信号重新触发父智能体处理新的子交付。
- **FR-024**: 父智能体的唤醒机制 MUST 保护父 durable 历史的稳定性；系统 MUST NOT 通过向 durable 事件流写入 `ReactivationPrompt` 一类机制性 `UserMessage` 或变化的交付详情消息来驱动下一轮处理。
- **FR-025**: 子交付详情 MUST 以一次性结构化输入参与父智能体的下一轮上下文构建，并在消费后清除；父 durable 历史只保留可追溯的子交付边界事实。
- **FR-026**: 一次性结构化输入 MUST 只承担运行时桥接职责，不参与 durable 回放或崩溃恢复；durable 回放只依赖边界事件、子会话入口与其他 durable 真相。
- **FR-027**: 即使一次性交付输入丢失，系统仍 MUST 通过持久化边界事实与子会话入口保留交付的可追溯性；父唤醒机制本身不作为 durable 业务事实落盘。
- **FR-028**: 父侧默认视图 MUST 以摘要投影方式展示子智能体状态、最近进展和最终结论。
- **FR-029**: 子会话入口、终态和可读时间线 MUST 独立持久化，不受父侧折叠或展开状态影响。
- **FR-030**: 在多个子智能体并发运行、恢复或交付时，系统 MUST 保证 lineage 查询、交付处理和父侧投影只作用于目标子智能体，不出现串扰、误关联或重复消费。
- **FR-031**: 缓存复用判断 MUST 基于 `runtime-prompt` fingerprint 体系计算的强指纹或等价机制；规格只约束可观察失效行为，不重造独立 key 规则。该强指纹 MUST 至少覆盖规范化工作目录、活动配置、工具集合、规则内容、prompt declarations、skills、工具元数据与上下文构建版本等会影响 prompt 构建结果的输入；任一相关输入变化都视为不同缓存键。
- **FR-032**: 当父智能体正在处理某轮工作时收到新的子交付，系统 MUST 将各次交付独立缓冲，并在父智能体可继续处理时可靠送达；不得丢失、合并不同子智能体的交付，或让同一交付被重复消费。
- **FR-033**: 最近活动的语义筛选与摘要化 MUST 优先采用确定性规则，例如角色优先级、预算裁剪和工具输出摘要；除非规格另行声明，不得仅为了构建 recent tail 再引入额外推理回合。
- **FR-034**: 当 lineage 关系不一致导致恢复无法安全继续时，系统 MUST 通过父侧可见的边界错误事实与系统诊断记录双通道暴露该问题，并阻止本次恢复继续。
- **FR-035**: 当子智能体被主动取消、强制终止或其他非成功方式结束时，系统 MUST 仍向父侧写入明确的终态边界事实和可读摘要，避免父侧留下无结论的悬挂任务。
- **FR-036**: 系统 MUST 保留 `SubRun` 与 `ChildSessionNode` 的职责分离：`SubRun` 表示单次 spawn 或 resume 触发的执行生命周期，`ChildSessionNode` 表示稳定 durable 的父子会话归属节点；系统 MUST NOT 将两者合并为单一概念。

### Key Entities *(include if feature involves data)*

- **父会话**: 发起子任务、接收边界事件和结果摘要的主会话实体。
- **子会话**: 承载单个子智能体 durable 真相、内部历史和恢复信息的独立会话实体。
- **ChildSessionNode**: 父侧 durable 的稳定子会话节点，表示子会话归属、入口和可恢复状态，不等同于某一次具体执行。
- **子执行实例（`SubRun`）**: 同一子会话的一次具体 spawn 或 resume 执行尝试，用于承载 started/finished 生命周期与执行统计。
- **边界事件**: 父侧历史中保留的子任务启动、完成、失败、恢复和交付事实。
- **lineage 关系**: 父会话、`ChildSessionNode`、子会话和子执行实例之间的可查询归属关系。
- **继承背景**: 父智能体传给子智能体的结构化背景，由压缩摘要与最近活动等部分组成。
- **继承块**: 由 `PromptDeclaration` 注入到 prompt system blocks 的父背景块，和任务消息分离表达，并拥有独立更新与缓存边界。
- **共享上下文缓存**: 在安全条件下可被父子智能体复用的稳定上下文准备结果。
- **父侧摘要投影**: 父会话中对子智能体状态、进展和结论的简化表示。
- **一次性交付输入**: 只服务父智能体下一轮处理的运行时结构化输入，不进入 durable 历史。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 在连续创建至少 10 个子智能体的验证场景中，100% 的新子智能体都生成独立子会话，且父历史中不出现来自子会话的 `AssistantFinal`、`ToolCall`、`ToolCallDelta`、`ToolResult`、`PromptMetrics`、`TurnDone` 等内部事件类型。
- **SC-002**: 在至少 10 个恢复验证场景中，100% 的恢复都沿用原子会话身份并生成新的子执行实例（`SubRun`），不出现恢复退化为重新创建并列子会话的情况。
- **SC-003**: 在支持缓存指标的 provider 上，针对规范化工作目录、活动配置名称、排序后的允许工具集合、规则文件内容摘要和上下文构建版本保持一致的重复启动验证场景，后续子智能体启动的 `cache_creation_input_tokens` 较首次下降至少 70%，以 `PromptMetrics` 事件日志为准；不支持该字段的 provider 必须在本 feature 交付前补齐等价缓存可观测指标，否则暂不纳入本条验收范围。
- **SC-004**: 在“内容不同但表面规模相似”的缓存安全测试中，100% 的测试都不会出现错误缓存复用。
- **SC-005**: 在父背景继承验证场景中，100% 的子智能体首条任务消息都只包含任务目标和直接上下文，不包含父背景全文拼接。
- **SC-006**: 在包含长工具输出的最近活动验证场景中，100% 的继承背景都以语义摘要形式传递，不出现无界原文继承。
- **SC-007**: 在“父 turn 已结束、多个子智能体随后交付”的验证场景中，100% 的子交付都能触发父智能体继续处理，且父 durable 历史不出现携带交付详情的可变任务消息或 `ReactivationPrompt` 类型的机制性 `UserMessage`。
- **SC-008**: 在重启恢复验证场景中，100% 的父子关系都能通过 durable 历史和 lineage 关系还原到正确子会话；若关系不一致，系统会明确暴露而不是静默伪造。

## Assumptions

- 本次规格聚焦于子会话边界、缓存契约和恢复语义，不扩展新的用户级协作能力集合。
- 旧的共享写入历史不属于本 feature 的支持范围；若仍存在 legacy 数据，系统将显式拒绝读取或恢复，并要求外部升级或清理流程处理。
- 父侧对运行中子智能体的实时观测主要依赖运行时能力；进程重启后允许实时观测丢失，但不允许 durable 真相丢失。
- 父子之间共享的是安全可复用的上下文准备结果，而不是跨请求直接共享外部推理请求的内部缓存。
- 子会话的保留与删除策略沿用现有会话生命周期规则；本次规格不额外引入新的 TTL、归档或级联删除政策。
