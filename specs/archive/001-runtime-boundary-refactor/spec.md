# Feature Specification: Runtime Boundary Refactor

**Feature Branch**: `[001-runtime-boundary-refactor]`  
**Created**: 2026-04-07  
**Status**: Draft  
**Input**: User description: "重构 runtime、session、agent 与 subrun 协议边界，明确 durable/live 真相、消除重叠 façade，并把已确认的高价值职责混乱点收敛为可执行规格"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Durable Child Execution Truth (Priority: P1)

作为维护运行时与调试线上问题的开发者，我希望子执行的归属关系、父子层级和触发来源在执行结束后仍能从历史记录中稳定重建，这样历史查询、回放、排障和过滤行为就不会依赖当前进程里是否还保留着 live 状态。

**Why this priority**: 这是当前最大的不确定源。只要历史真相和 live 真相不一致，后续所有查询层、过滤层和前端展示都会持续争论“哪个才是真的”。

**Independent Test**: 可以单独通过“启动子执行、等待其结束、清空 live 状态、再执行历史查询与事件回放”的方式验证；如果仍能得到一致的父子关系和来源信息，这一层就成立。

**Acceptance Scenarios**:

1. **Given** 一个子执行已经完成且 live 控制状态已不存在，**When** 调用历史状态查询，**Then** 返回的子执行标识、父级关系、层级深度和触发来源与执行期间观测到的信息一致。
2. **Given** 一个历史会话中包含多个同父子执行，**When** 对同一份历史重复回放或过滤，**Then** 每次得到的归属关系和作用域结果保持一致。

---

### User Story 2 - Clear Runtime Responsibilities (Priority: P2)

作为后端贡献者，我希望 session 生命周期、单次执行主循环、多 Agent live 控制和执行编排拥有清晰且单向的职责边界，这样我在修改某一层时不需要先理解另一层里重复存在的同类逻辑。

**Why this priority**: 当前最大的维护成本来自重叠 surface 和环形调用。即使功能不坏，也会让任何后续改动变成高风险操作。

**Independent Test**: 可以单独通过边界设计评审验证；如果评审者能从一份边界设计中明确看到每个边界的职责、依赖方向、替换路径和删除前提，这个故事就成立。

**Acceptance Scenarios**:

1. **Given** 一项执行链路相关改动需要删除旧 surface，**When** 评审迁移方案，**Then** 能明确看到当前调用方、替代入口和删除顺序，而不需要额外追溯聊天记录。
2. **Given** 一个边界被声明为会话真相边界或执行编排边界， **When** 检查该边界的职责说明， **Then** 不会再出现两个边界同时声称拥有同一类核心职责。

---

### User Story 3 - Consistent Query and Scope Behavior (Priority: P3)

作为消费会话历史、子执行状态和范围过滤能力的上层调用方，我希望同一份执行数据在历史回放、增量事件和范围过滤中使用一致的归属语义，这样前端和服务端就不需要分别维护两套推断规则。

**Why this priority**: 一致的查询和过滤语义是把协议改动真正转化为可用产品行为的必要条件，否则“协议补齐了字段”仍然无法减少上层复杂度。

**Independent Test**: 可以单独通过历史回放、增量订阅和范围过滤三条路径对同一执行样本做交叉比对；三条路径返回的归属和作用域一致，即为通过。

**Acceptance Scenarios**:

1. **Given** 一个包含根执行、直接子执行和孙级子执行的会话历史，**When** 分别请求自身、直接子级和整棵子树范围，**Then** 每种范围都只返回与其定义一致的事件集合。
2. **Given** 不同工作目录下存在不同的 agent 定义，**When** 从这些上下文发起执行，**Then** 每次执行都解析到与目标工作上下文一致的定义，而不是静默退回到进程级默认值。

---

### Edge Cases

- 历史数据缺少新的子执行归属元数据时，系统必须定义并暴露降级行为，而不是伪造看似精确但实际上推断出来的完整层级。
- 一个父执行在子执行仍然运行时结束、取消或崩溃，历史查询与范围过滤仍需给出可解释的一致结果。
- 同一父执行在一轮中发起多个子执行时，系统不能把这些并发子执行错误地归并成同一个所有者。
- 子执行使用独立会话存储时，其归属语义必须与共享会话模式保持一致，不能因为落盘位置不同而改变父子关系定义。
- 当 live 状态与历史状态暂时不一致时，系统必须明确哪一层负责最终真相，避免同一个查询在重试后返回相互冲突的结果。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST 为每个新产生的子执行记录稳定的归属元数据，至少能表达子执行标识、父执行回合、父子执行关系和层级深度。
- **FR-002**: System MUST 在子执行由工具触发时记录稳定的触发来源关联，使同一子执行生命周期可以与触发它的工具调用建立一对一链路。
- **FR-003**: System MUST 允许在子执行已经结束且 live 控制状态已清理后，仍能仅依靠历史记录重建该子执行的状态、层级和触发来源。
- **FR-004**: System MUST 将历史记录定义为历史子执行查询的主事实源，并只在当前仍在运行的场景下用 live 状态补充运行态信息。
- **FR-005**: System MUST 为会话生命周期、单次执行主循环、多 Agent live 控制和执行编排定义互不重叠的职责边界。
- **FR-006**: System MUST 为提交执行、回放历史、中断执行和查询子执行状态提供唯一且明确的公共入口，不允许长期保留语义重叠的双轨 surface。
- **FR-007**: System MUST 为每一个计划删除的旧入口提供迁移阶段说明，至少包含当前调用方、替代入口、删除前提和推荐顺序。
- **FR-008**: System MUST 将执行归属与存储位置解耦，使共享会话与独立会话两种模式暴露相同的父子执行语义。
- **FR-009**: System MUST 为子执行事件范围过滤提供稳定的“自身”“直接子级”“整棵子树”语义，并使用显式归属关系而不是仅靠事件顺序推断。
- **FR-010**: System MUST 明确定义旧历史数据的兼容策略，包括哪些归属字段可能为空、如何降级展示以及何时允许返回不完整结果。
- **FR-011**: System MUST 让执行上下文中的 agent 定义解析与刷新行为跟随目标工作上下文，而不是只依赖进程启动时的全局快照。
- **FR-012**: System MUST 将本次重构的架构文档拆分为“已确认事实”“协议改动设计”“执行边界设计”三类独立产物，以便评审者分别接受或拒绝不同层次的内容。
- **FR-013**: System MUST 让历史回放、增量事件和子执行状态查询共享同一套子执行身份与层级语义，避免上层调用方维护多套归因规则。

### Key Entities *(include if feature involves data)*

- **Child Execution Lifecycle Record**: 表示一次子执行从启动到完成的稳定身份与归属事实，包含子执行身份、父级关系、层级和触发来源。
- **Execution Ownership Record**: 表示一次执行属于谁、由谁触发、处于哪一层父子树中，用于统一根执行和子执行的归属语义。
- **Responsibility Boundary**: 表示系统中的一个明确职责区，定义其拥有的核心职责、允许依赖的上游边界和禁止承担的职责。
- **Migration Stage**: 表示一次可独立评审和交付的重构阶段，包含目标、入口替换关系、删除前提和验收标准。
- **Event Scope Definition**: 表示会话事件过滤中的作用域定义，用来区分仅自身、直接子级和整棵子树的可见范围。
- **Execution Resolution Context**: 表示一次执行所依赖的工作上下文，用于决定可见的 agent 定义和与该上下文绑定的执行策略。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 对于所有在该特性上线后创建的子执行，历史状态查询在 live 状态被清理后仍能返回与运行期间一致的子执行标识、父级关系、层级深度和触发来源。
- **SC-002**: 评审者可以仅依赖三份独立架构产物完成事实核对、协议评审和边界评审，而不需要回看聊天记录或自行补全上下文。
- **SC-003**: 每个迁移阶段都能明确指出提交执行、回放历史、中断执行和状态查询的唯一入口，且评审中不再出现“旧入口删掉后谁接手”的未决问题。
- **SC-004**: 同一份包含嵌套子执行的历史在重复回放、范围过滤和状态查询时，得到的自身、直接子级和整棵子树结果保持一致。
- **SC-005**: 从不同工作上下文触发的执行在验收样本中都能解析到与其上下文一致的 agent 定义，不再出现被无提示地替换成全局默认定义的情况。

## Assumptions

- 现有会话历史回放和增量事件传输仍然是上层消费执行事实的主路径，本次规格不引入第二套主线协议。
- 共享会话模式仍是当前默认执行路径，独立会话模式继续保留但必须遵守同一套归属语义。
- 仓库中已经存在旧历史数据，因此本次重构必须定义历史兼容行为，但不要求为旧数据补写缺失事实。
- 对内部入口和模块边界做破坏性调整是可接受的，前提是迁移顺序、替代入口和兼容边界被显式写入设计产物。
- 上层调用方可以逐步采用新增的归属信息，只要新增语义在历史回放、增量事件和状态查询中保持一致。
