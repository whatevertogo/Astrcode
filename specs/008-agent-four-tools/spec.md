# Feature Specification: Astrcode Agent 协作四工具重构

**Feature Branch**: `[008-agent-four-tools]`  
**Created**: 2026-04-11  
**Status**: Draft  
**Input**: User description: "将 Astrcode 的 agent 协作公开面收敛为 spawn、send、observe、close 四个工具，并基于现有 runtime 与 event log 重构生命周期、durable mailbox、父子唤醒与快照语义。"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 父 Agent 管理可复用的子 Agent (Priority: P1)

作为运行时维护者，我希望父 Agent 能创建一个子 Agent、在它完成一轮工作后继续向它发送后续指令、并在任务结束时显式关闭它，这样协作模型才会和四工具公开面保持一致，不需要为每次返工重复创建子 Agent。

**Why this priority**: 这是四工具模型能否成立的基础。如果子 Agent 不能在完成一轮后继续存活，`send(agentId, message)` 的核心价值就不存在。

**Independent Test**: 仅实现这一故事时，可以通过创建子 Agent、让其完成一轮、再次发送新指令、观察其回到可接收状态并最终关闭，独立验证四工具模型的核心价值。

**Acceptance Scenarios**:

1. **Given** 父 Agent 已创建一个子 Agent，**When** 子 Agent 完成当前一轮工作，**Then** 子 Agent 必须保持可寻址并进入空闲可继续接收指令的状态，而不是被视为已结束不可再用。
2. **Given** 一个处于空闲状态的子 Agent，**When** 父 Agent 发送新的后续指令，**Then** 系统必须接受该指令并驱动子 Agent 开始下一轮工作。
3. **Given** 父 Agent 已确认子 Agent 的任务已经完成，**When** 父 Agent 执行关闭操作，**Then** 该子 Agent 及其后代必须被终止，之后不再接受新的协作消息。

---

### User Story 2 - 父子消息在异步和重启场景下保持可恢复 (Priority: P2)

作为运行时维护者，我希望父子 Agent 之间的消息在父忙碌、子忙碌或运行时重启时仍然可恢复、可重放、不会静默丢失，这样异步协作才具备可预期的可靠性。

**Why this priority**: 四工具公开面去掉了显式等待工具后，系统必须靠消息排队与唤醒来保证协作连续性；如果消息在边界场景下丢失，整个模型会变得不可依赖。

**Independent Test**: 仅实现这一故事时，可以通过发送消息、制造忙碌和重启场景、检查未确认消息是否仍会出现在后续处理流程中，独立验证消息可靠性。

**Acceptance Scenarios**:

1. **Given** 目标 Agent 正在处理当前一轮工作，**When** 另一端向它发送新消息，**Then** 新消息必须被排队等待下一轮处理，而不能插入当前正在执行的上下文。
2. **Given** 一批已进入待处理队列但尚未完成确认的消息，**When** 运行时在处理过程中重启，**Then** 这些消息必须能够在恢复后重新进入待处理集合，而不是静默消失。
3. **Given** 子 Agent 向父 Agent 发送消息且父 Agent 当前空闲，**When** 消息成功入队，**Then** 父 Agent 必须在下一个可用轮次被唤醒处理该消息。

---

### User Story 3 - 维护者获得简化且可观测的协作接口 (Priority: P3)

作为运行时维护者，我希望公开协作接口只保留四个职责清晰的工具，并能通过 `observe` 获取可靠快照，这样可以降低 prompt 心智负担、减少工具歧义，并提升故障定位效率。

**Why this priority**: 这是对外体验和长期维护成本的直接改进，但不如核心生命周期与消息可靠性更基础，因此排在 P3。

**Independent Test**: 仅实现这一故事时，可以通过检查工具目录、调用观测接口、验证访问边界和快照字段，独立验证公开面是否真正被收敛且可用。

**Acceptance Scenarios**:

1. **Given** 系统已完成重构，**When** 维护者查看公开协作工具目录或文档，**Then** 只应看到 `spawn`、`send`、`observe`、`close` 四个协作工具。
2. **Given** 父 Agent 需要判断子 Agent 当前状态，**When** 父 Agent 调用 `observe`，**Then** 系统必须返回包含生命周期状态、最近一轮结果、待处理消息数量、活动任务摘要、待处理任务摘要以及最近输出摘要的 observe 结果。
3. **Given** 非直接父级或无关 Agent 尝试观测目标 Agent，**When** 调用 `observe`，**Then** 系统必须拒绝该请求。

---

### Edge Cases

- 当消息发送到已终止的 Agent 时，系统必须明确拒绝该操作，且不得入队、不得触发唤醒。
- 当某轮处理刚开始后又收到新消息时，新消息必须保留到下一轮，而不是混入当前轮的上下文。
- 当系统在消息批次开始处理后、完成确认前崩溃时，同一批消息重新出现必须被视为允许的重放，而不是静默丢失或伪装成全新消息。
- 当子 Agent 在发送消息后被关闭时，父 Agent 后续看到“发送方已终止但消息仍存在”必须被视为合法状态。
- 当对不存在的 Agent 或已经关闭完成的 Agent 再次执行 `close` 时，系统必须将其视为幂等关闭请求，并返回可审计的关闭结果，而不是制造新的错误分支。
- 历史上的共享会话子 Agent 数据仍需可读，但新的子 Agent 不再继续生成这种写法。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 仅公开四个协作工具，其 LLM 注册名称为 `spawn`、`send`、`observe`、`close`，不再保留旧工具名。
- **FR-002**: 系统 MUST 从公开面彻底移除原有的等待、上行交付、显式恢复和旧命名的发送/关闭工具，不保留兼容公开入口。
- **FR-003**: 系统 MUST 为每个根 Agent 和子 Agent 维护统一的父子控制关系，使根 Agent 与子 Agent 位于同一棵可验证的控制树中。
- **FR-004**: 系统 MUST 让新创建的子 Agent 在完成单轮工作后进入可继续接收消息的空闲状态，而不是自动终止。
- **FR-005**: 系统 MUST 将 Agent 的生命周期状态与最近一轮执行结果分开表达，并允许观察接口同时返回二者。
- **FR-006**: 系统 MUST 仅允许直接父级与直接子级之间互相发送消息，并 MUST 拒绝发送到已终止 Agent、兄弟 Agent、越级 Agent、跨树 Agent 的消息请求。
- **FR-007**: 系统 MUST 在目标 Agent 空闲时，允许新消息触发其下一轮处理；若目标正在运行，则消息必须排队到下一轮。
- **FR-008**: 系统 MUST 在 Agent 开始某一轮处理时，通过 snapshot drain 只消费该轮开始瞬间已经存在的一批待处理消息，并将轮中新增消息保留到下一轮。
- **FR-009**: 系统 MUST 为每条协作消息分配稳定的唯一标识（`delivery_id`），并在后续处理与观测中保留该标识。
- **FR-010**: 系统 MUST 持久化所有尚未确认的协作消息，使其在运行时重启后仍可恢复为待处理消息。
- **FR-011**: 系统 MUST 采用 at-least-once 投递语义，并在 snapshot drain 阶段基于 `delivery_id` 去重，确保同一消息不会被同一 Agent 的同一轮次重复处理。
- **FR-012**: 系统 MUST 在消息被排队时记录发送方当时的生命周期状态、最近一轮结果以及可打开会话目标，并在后续注入或观察时继续使用这些快照值。
- **FR-013**: 系统 MUST 提供 `observe` 结果，至少包含目标 Agent 身份信息、生命周期状态、最近一轮结果、会话目标、阶段、轮次数、待处理消息数量、活动任务摘要、待处理任务摘要和最近输出摘要。
- **FR-014**: 系统 MUST 仅允许直接父级调用 `observe` 查看其直接子级。
- **FR-015**: 系统 MUST 将 `close` 定义为唯一终止手段：终止目标 Agent 及其后代、丢弃其未确认的 mailbox 消息、取消运行中的 turn、清理 pending wake item，并阻止后续消息继续进入。
- **FR-016**: 系统 MUST 将新的子 Agent 持久化为独立会话实体，同时继续允许系统读取历史共享会话数据。
- **FR-017**: 系统 MUST 保留面向界面和时间线的子会话通知能力，但这些通知不得再承担父子协作主通道职责。
- **FR-018**: 系统 MUST 在提示词、工具描述和示例中统一传达四工具心智模型，不再引导模型使用已移除的旧工具或阻塞式等待。
- **FR-019**: 系统 MUST 在提示材料中说明 `delivery_id` 的稳定性，使模型能够识别因异常恢复而重新出现的消息，并将其视为上一轮的延续而非新任务。

### Key Entities *(include if feature involves data)*

- **Agent Instance**: 一个可被父 Agent 创建、发送消息、观察和关闭的持久协作对象，具有身份、父子关系、生命周期状态、最近一轮结果和可打开会话目标。
- **Agent Mailbox Message**: 一条由父子 Agent 之间发送的协作消息，具有稳定标识、发送方与接收方身份、消息正文以及排队时的发送方状态快照。
- **Mailbox Batch**: 某个 Agent 在单轮处理开始时一次性接管的固定消息集合，具有批次标识和一组稳定消息标识。
- **Observe Result**: `observe` 返回的目标 Agent 查询结果，汇总实时控制状态、对话投影状态以及 mailbox 派生摘要。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 在公开工具注册表、文档描述和 prompt 指导中，协作工具数量固定为 4 个，且不再出现旧协作工具名称。
- **SC-002**: 在验证场景中，一个子 Agent 完成第一轮工作后，能够在不重新创建实例的前提下成功接收并执行第二条新指令。
- **SC-003**: 在异常恢复验证中，所有未确认的待处理消息要么被重新纳入待处理集合，要么因显式关闭而被标记丢弃，不允许出现静默丢失。
- **SC-004**: 在并发消息验证中，正在运行的 Agent 当前轮只处理轮开始时的固定消息批次，轮中新增消息 100% 延迟到下一轮。
- **SC-005**: 在权限验证中，直接父级调用 `observe` 能获得完整快照，非直接父级调用 `observe` 100% 被拒绝。

## Assumptions

- 本次重构按干净切换执行，旧协作工具不再保留公开兼容入口。
- 根 Agent 会被提升为一等控制对象，使其与子 Agent 使用同一套父子树语义。
- 新创建的子 Agent 统一使用独立会话；共享会话仅保留为历史数据读取能力。
- 协作消息采用至少一次投递语义，重复投递通过稳定标识识别，而不是追求严格的一次且仅一次。
- `close` 当前只支持整棵子树终止，不支持仅关闭单节点并保留其后代继续运行。
- `resume` 本轮仅保留为内部未来扩展预留，不出现在任何公开工具、文档或提示词中。
