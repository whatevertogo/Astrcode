# Feature Specification: 删除死代码与兼容层收口

**Feature Branch**: `006-prune-dead-code`  
**Created**: 2026-04-10  
**Status**: Draft  
**Input**: User description: "重新定义删除死代码、无意义代码和向后兼容层的清理需求，覆盖前端路由/API、后端未消费端点、legacy 子智能体兼容路径和相关文档/测试收口。"

## Scope Overview

本特性不是一次泛泛的“代码整理”，而是一次明确的收口动作：

1. 删除当前产品路径没有消费、只有测试或文档还在自我引用的前端读模型、状态投影和后端接口。
2. 删除已经失去主线地位、仅为旧子智能体/旧子会话/旧 sub-run 语义保留的向后兼容层。
3. 保留当前仍有真实消费者的核心摘要语义，例如子 Agent 终态交接摘要与父侧通知摘要；删除没有 UI 或运行时消费者的重复摘要投影。
4. 同步收口文档、测试、夹具和术语，避免仓库继续暗示“不再支持但好像还能用”的表面能力。
5. 除了删除无人消费 surface，本特性还覆盖 subrun 领域里“虽然还在使用、但语义重复或归属错误”的核心模型与服务契约收口，例如重复状态模型、重复接受结果模型、字段归属错误和控制边界错位。

## Clarifications

### Session 2026-04-10

- Q: 这次 spec 是否要把 subrun 核心模型与服务契约冗余清理也纳入范围，而不只做前后端死代码和 legacy surface 清理？ → A: 纳入。将候选项 1-6 视为当前 feature 的核心冗余清理范围，7-8 作为顺手修正的注释契约一起纳入。
- Q: 这次是否要把 `SubRunDescriptor`、optional `parent_turn_id` 和 descriptorless / legacy downgrade 的公开读模型一起做彻底 cutover，而不是继续在 protocol/server/frontend 保留半套兼容语义？ → A: 是。删除 `SubRunDescriptor`，将 `parent_turn_id` 收口为必填，并让 descriptorless / legacy downgrade 的公开读模型退出主线。
- Q: 当 `SubRunOutcome` 合并进 `AgentStatus` 后，`TokenExceeded` 是否要作为正式公开终态保留，而不是折叠成 `Completed`？ → A: 保留。`TokenExceeded` 作为 `AgentStatus` 的正式终态，在 core/runtime/protocol/frontend 全链路可见。
- Q: 若后续还有未穷尽的冗余或归属选择，本 feature 的默认决策原则是什么？ → A: 默认选择最干净、最少兼容、最易理解的收口方案；除非某能力仍是当前主线不可替代的一部分，否则不为兼容保留额外壳层、派生字段或双轨模型。
- Q: `ChildSessionNotification.open_session_id` 与 `child_ref.open_session_id` 重复时，本次应保留哪一个作为正式打开目标事实？ → A: 只保留 `child_ref.open_session_id`。通知层和 protocol DTO 不再额外重复存放 `open_session_id`，避免同一 child open target 出现双份可漂移字段。
- Q: child/session 相关 protocol DTO 的 `status` 是否继续允许 `String` 弱类型暴露？ → A: 否。protocol 必须提供独立的强类型状态枚举并与 core 正式状态一一映射，前端不再依赖字符串匹配理解 child/subrun 状态。
- Q: `HookCompactionReason` 与 `CompactTrigger` 的重叠语义应如何收口？ → A: `Reactive` 只保留为 hook/runtime 内部原因；一旦落到 durable compaction trigger，就显式映射到唯一正式 trigger，并把该映射集中在单一 owner，而不是散落在多处调用点。
- Q: `PromptMetrics` 在 storage event、agent event 和 protocol event 三层逐字段重复时，本次是否只做搬运保持现状？ → A: 否。提取共享 payload 结构并在三层复用，避免同一组指标字段继续以三份 variant 载荷长期并存。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 收口正式支持面 (Priority: P1)

作为维护者，我希望仓库里只保留当前产品确实在使用、并且有人负责的前后端表面能力，这样我在阅读代码和推进后续重构时，不会被“看起来存在、实际上没人用”的接口、投影和状态模型误导。

**Why this priority**: 如果正式支持面不先收口，后续每一次重构、排障和规格更新都会继续背着伪能力和伪边界前进，复杂度会反复回弹。

**Independent Test**: 对前端导航/读模型、后端接口和摘要语义做一次消费者审计；只实现这一项，也能立即让当前代码库的“正式支持范围”变得清晰、可验证。

**Acceptance Scenarios**:

1. **Given** 某个前端 API、投影函数或导航辅助结构没有任何当前 UI 或交互流程消费，**When** 完成本次清理，**Then** 该能力连同其配套引用一起被删除，而不是继续作为“以后可能会用”的预实现保留。
2. **Given** 某个后端接口没有当前产品入口、没有活跃消费者，也没有被确认仍是正式支持的外部契约，**When** 完成本次清理，**Then** 该接口被移出正式支持面，而不是继续作为空壳或未收口接口留在代码库中。
3. **Given** 同一语义存在多个摘要表达层，**When** 完成本次清理，**Then** 只有当前 UI 或运行时真正消费的摘要事实被保留，重复投影或无人消费的摘要层被删除。

---

### User Story 2 - 删除旧模型兼容层 (Priority: P1)

作为维护者，我希望系统不再为了旧子智能体模型、旧会话写入方式或旧控制路径保留双轨逻辑，这样运行时、协议和前端读模型都只围绕当前主线能力演进，而不是继续对旧世界做显式照顾。

**Why this priority**: 向后兼容层会把已经结束的迁移期永久固化成维护成本，尤其会污染运行时边界、状态类型、错误分支和测试夹具。

**Independent Test**: 只做这一项，也能让系统从“主线能力 + 兼容层”的双轨结构收束为“一个主线 + 明确失败边界”的单轨结构。

**Acceptance Scenarios**:

1. **Given** 某条读写、状态查询、取消或导航路径仅用于兼容旧子智能体/旧 sub-run/旧共享历史语义，**When** 完成本次清理，**Then** 该兼容路径被删除，而不是继续以 legacy、downgrade 或 fallback 的形式长期保留。
2. **Given** 某条旧路径仍然是当前活跃用户流程的唯一入口，**When** 进行清理，**Then** 系统先把该活跃流程迁移到唯一的正式路径，再删除旧路径，而不是直接留下功能缺口。
3. **Given** 遇到历史遗留数据或旧调用方式，**When** 这些输入落入当前主线流程，**Then** 系统宁可明确失败，也不能继续伪造 lineage、摘要、导航关系或“部分可用”的兼容视图。

---

### User Story 3 - 文档、测试与术语同步收口 (Priority: P2)

作为维护者，我希望当前文档、测试和夹具只描述仍被支持的能力与边界，这样后来者不会再根据过时 spec、骨架路由测试或 legacy 术语，把已经决定删除的东西重新扩回来。

**Why this priority**: 如果代码删了但 spec、开放项、测试夹具和术语没删，仓库会继续对人和自动化工具发出错误信号，等于把死代码以另一种形式留下来。

**Independent Test**: 只做这一项，也能让当前文档与测试基线重新对齐主线架构，为后续计划、实现和 review 提供稳定边界。

**Acceptance Scenarios**:

1. **Given** 当前文档把无人消费或已决定删除的表面能力描述成 stable、experimental 或待收口接口，**When** 完成本次清理，**Then** 当前文档不再继续宣传这些能力。
2. **Given** 某些测试、夹具或断言只是在证明已删除路径“还存在”，**When** 完成本次清理，**Then** 这些测试和夹具被删除或改写为围绕保留能力的验证。
3. **Given** 清理后的仓库仍保留若干公共表面能力，**When** 维护者查看当前规格，**Then** 每一项保留能力都能找到清晰用途、消费者类型和所属边界，而不是靠历史上下文猜测其存在理由。

### Edge Cases

- 某个接口当前没有前端消费者，但确实仍承担正式的外部契约职责；如果不能明确其拥有者、用途和存续理由，就应按死代码处理。
- 某个 legacy 路径看起来只是兼容层，但当前活跃流程仍通过它完成关键动作；此时必须先迁移当前流程，再删除旧路径。
- 同一个“summary”字段在不同层次出现时，必须区分“当前仍被消费的摘要事实”和“仅剩重复投影的摘要外壳”，避免误删核心能力。
- 某些类型或 trait 虽然仍被当前主线使用，但如果它们只是另一套重复语义或错误归属的壳，本特性也应收口，而不是因为“仍有调用方”就自动保留。
- 某个状态值如果承载真实业务语义，即使它目前被部分层级折叠映射，本特性也应把它收口成单一正式状态，而不是继续保留“内部一个状态、外部另一个状态”的双轨表达。
- 某个描述符或可选字段如果只是为了维持旧 downgrade 公共语义而存在，本特性应优先把它收口到必填核心字段或直接删除，而不是继续在公开合同里保留“也许没有”的半兼容形态。
- 若某个字段、DTO 或 helper 只是为了让调用方“少改一点”而存在，但并不承载不可替代的主线语义，本特性默认删除它而不是继续包装兼容层。
- 某个 child open target 在嵌套引用与外层通知里各存一份时，即使今天两者值相同，也应视为重复事实源；本特性必须收口为单一来源，避免后续写入或映射时发生漂移。
- 若 protocol DTO 把核心状态降成字符串，即使当前前端还能靠字符串匹配工作，也应视为正式契约退化；本特性需要恢复为枚举型协议事实，而不是继续容忍弱类型外泄。
- 若 hook 侧与 durable event 侧都在表达“压缩触发原因”，但粒度不同，本特性必须明确哪个枚举负责内部控制、哪个枚举负责持久化事实，以及它们之间的唯一映射；不能依赖调用点各自猜测。
- 某组指标数据如果在 storage、domain event 与 protocol event 中逐字段复制，即使当前字段值保持一致，也应视为结构性重复；本特性应把它收口为共享 payload，而不是继续接受三层搬运。
- 旧数据、旧客户端或旧测试在兼容层删除后可能直接失败；系统可以不保证继续可用，但不能继续伪造关系、拼接假摘要或输出误导性的部分结果。
- 当前文档可以保留归档材料作为历史记录，但当前有效规格、开放项和实现说明不能再把已删除能力写成仍在维护的现状。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 先在 `findings.md` 中刷新并验证清理清单，把候选表面能力划分为”立即删除””先迁移后删除””明确保留”三类，并以当前真实消费者与责任边界作为判定依据。
- **FR-002**: 任何没有当前产品消费者、没有明确拥有者、也没有被确认仍属正式支持契约的前端或后端表面能力 MUST 在本次清理中删除。这包括仅为未来可能性预埋的空壳接口、骨架路由、占位投影和未落地控制面。
- **FR-003**: 当前仍被运行时或 UI 消费的核心摘要事实 MUST 保留；没有消费者的重复摘要投影、重复摘要 API 或平行摘要读模型 MUST 删除。
- **FR-004**: 前端导航相关代码 MUST 只保留当前状态驱动导航实际需要的状态、过滤与投影；没有真实消费的路由化抽象、辅助读模型或子视图投影 MUST 删除。
- **FR-005**: 后端对外暴露的接口 MUST 只保留当前正式支持的能力；无人消费、无人认领或只剩测试自证的接口 MUST 从正式支持面移除。
- **FR-006**: 已删除的表面能力 MUST 不再仅以类型、注释、文档、测试、夹具或错误码的形式残留在主线代码中。
- **FR-007**: 仅为旧子智能体模型、旧共享历史语义、旧状态降级语义或旧控制路径服务的向后兼容层 MUST 从运行时、协议、服务端和前端读模型中同步移除。
- **FR-008**: 对已经决定不再支持的旧输入（包括 descriptorless subrun、legacyDurable source、旧共享历史语义），系统 MUST 停止提供伪完整的兼容展示；若这些输入进入主线流程，系统 MUST 以明确失败替代猜测性恢复、猜测性 lineage 或猜测性摘要。明确失败的错误响应 MUST 保留原始上下文（如哪些字段缺失、哪个旧语义不被支持），MUST NOT 丢弃根因信息（宪法 VII）。
- **FR-009**: 若某个 legacy 风格的接口仍是当前活跃流程的唯一入口，系统 MUST 在同一次特性中提供唯一的正式替代路径，并将活跃流程切换完成后再删除旧入口。具体地，当前”取消子会话”按钮必须从旧 cancel route 切换到 `closeAgent`，删除旧 cancel route 与 client wrapper。
- **FR-010**: 清理完成后，每个仍保留的关键用户动作 MUST 只有一条正式路径，不允许同时存在”主线入口”和”兼容入口”并行维护。
- **FR-011**: 当前有效文档 MUST 与清理结果一致，不得继续把已删除、无人消费或不再维护的能力描述为 stable、experimental、planned 或可选入口。
- **FR-012**: 当前测试基线 MUST 围绕保留能力和明确失败边界重建，不得继续为了维持已删除路径的存在感而保留专门测试。
- **FR-013**: 每个被明确保留的公共表面能力 MUST 在 `contracts/retained-surface-contract.md` 中说明其用途、消费者类型和所属边界，避免后来者只能通过历史提交推断其存在理由。
- **FR-014**: 本次清理除了 orphan surface 与 legacy surface 外，MUST 一并收口 subrun 领域中语义重复、状态重复、接受结果重复、字段归属错误或 trait owner 错位的核心模型与服务契约。
- **FR-015**: 对仍在使用但被判定为冗余的核心模型与服务契约，系统 MUST 以”合并到唯一语义模型”或”迁移到唯一 owner boundary”的方式收口，而不是继续长期并存。
- **FR-016**: 若 `SubRunDescriptor` 只是重复封装已有 subrun 身份信息，系统 MUST 删除该模型，并将其承载的必要字段直接收口到唯一正式字段，而不是继续通过额外 descriptor 对象公开表达。
- **FR-017**: `parent_turn_id` 若仍属于 subrun 核心身份与血缘事实，系统 MUST 将其收口为必填核心字段；不再为 descriptorless 或 legacy downgrade 公开语义保留 optional 形态。
- **FR-018**: descriptorless、legacy downgrade 或等价”部分可用” subrun 公开读模型 MUST 退出主线（本条是 FR-007 在 subrun 降级语义上的具体实例）；系统对不再支持的旧输入只保留明确失败（FR-008），不再保留半兼容读取合同。
- **FR-019**: subrun 终态 MUST 收口到单一正式状态模型 `AgentStatus`；如果 `TokenExceeded` 仍代表独立业务语义，系统 MUST 在 core、runtime、protocol 与 frontend 全链路将其作为正式终态保留，而不是折叠成其他状态。`SubRunOutcome` 的其他变体 MUST 显式映射到 `AgentStatus`：`Cancelled` 为唯一取消终态（`Aborted` 退出主线），`Completed` 保持现有语义。
- **FR-020**: 对未被逐项列出的同类冗余项（与已列出项共享相同模式：重复状态模型、重复 descriptor、字段归属错位、trait owner 错位），系统 MUST 默认采用最干净、最少兼容、最易理解的收口策略：保留唯一正式模型，删除重复壳层、派生兼容字段和双轨语义，除非它们仍是当前主线不可替代的一部分。

### Scope

- 本次清理覆盖前端状态导航面、前端会话/子会话 API 面、服务端 HTTP 面、运行时兼容分支、协议暴露面、subrun 核心模型、service contract / trait owner 以及与之对应的当前文档与测试资产。

### Invariant Requirements

- **FR-021**: 本次清理 MUST 不破坏当前仍在使用的主线能力，包括当前会话浏览、当前子执行浏览、当前子会话直开、当前配置读写、当前消息提交流程，以及当前活跃的子执行控制动作。
- **FR-022**: child session 的打开目标 MUST 只有一个正式事实源；若 `child_ref` 已承载 `open_session_id`，则通知、DTO 与其他外层载荷 MUST 不再重复持有同值字段。
- **FR-023**: protocol 层对 child/subrun 相关状态的公开表达 MUST 使用独立强类型枚举，而不是 `String`；前端与服务端映射 MUST 以正式状态枚举为边界，不再依赖字符串匹配。该枚举 MUST 为纯 DTO 镜像，不含运行时策略或默认推断（宪法 III）。
- **FR-024**: 对”压缩触发原因”这组语义，系统 MUST 明确区分内部运行时原因与 durable trigger；若 `Reactive` 不是 durable 正式枚举成员，则其落盘映射 MUST 只有一条集中定义的正式路径。
- **FR-025**: `PromptMetrics` 这类跨 storage event、agent event 与 protocol event 共享的指标数据 MUST 提取为可复用共享 payload，而不是在多个 enum variant 中长期维持逐字段重复定义。
- **FR-026**: `ChildAgentRef` 等核心/协议读模型 MUST 不再承载仅供 UI 使用且没有真实分支意义的硬编码布尔字段（如 `openable`）；调用方应基于正式身份与状态事实推导行为，而不是依赖永远为真的兼容标记。

### Key Entities *(include if feature involves data)*

- **正式支持面**: 当前产品或运行时真正依赖、且有明确拥有者与用途说明的前端状态、接口、协议或交互入口。
- **死代码表面能力**: 没有真实消费者，只剩类型、注释、测试、文档或占位实现证明其存在的能力。
- **先迁移后删除能力**: 设计上已经过时，但当前活跃流程仍暂时依赖，因此必须先切换到唯一替代路径再删除的能力。
- **兼容层工件**: 为旧子智能体模型、旧会话写入语义、旧状态降级展示或旧控制路径保留的额外类型、分支、接口、错误码或夹具。
- **摘要事实**: 当前仍被消费、用于表达子 Agent 终态交接或父侧通知的信息。
- **重复摘要投影**: 与现有摘要事实表达同一语义，但当前没有 UI 或运行时消费者的额外投影层。
- **核心模型冗余**: 虽仍被主线调用，但与另一套状态、描述符、接受结果或字段归属表达同一语义的重复领域模型。
- **归属错位契约**: 职责应属于某一边界，却暂时挂在其他 trait、DTO 或运行时结构上的接口与字段。
- **唯一打开目标事实**: child session / child agent 可直开的目标 session 标识，只允许在一处正式模型中承载，其他层只能引用或投影，不再重复存储。
- **协议状态枚举**: protocol 层为 child/subrun 状态提供的独立强类型 DTO 枚举，负责承接 core 正式状态并为前端提供类型安全边界。
- **压缩原因映射**: 从 hook/runtime 内部 compaction reason 到 durable trigger 的唯一归一规则，用来避免 `Reactive`/`Auto`/`Manual` 在多层各自解释。
- **共享指标载荷**: 被 storage event、agent event 和 protocol event 共同复用的 prompt metrics payload，避免同一组指标字段在多处重复声明。
- **清理清单**: 对所有候选表面能力给出的保留、迁移或删除决策集合，是实施和验证本特性的审计基线。

### Terminology

- **支持面（surface）**: 本文档统一使用"支持面"指代前后端公开能力集合；与 plan/tasks 中的"surface"同义，不再使用"表面能力"

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 在本次审计范围内，100% 的保留表面能力都能指向当前真实消费者与明确边界；不存在“没人用但先留着”的保留项。
- **SC-002**: 在本次审计范围内，前端导出 API/投影与后端公开接口中，0 个能力只剩测试或文档引用而没有当前产品消费者。
- **SC-003**: 在本次审计范围内，0 个仅为旧子智能体/旧共享历史/旧 sub-run 语义保留的兼容入口继续留在主线能力面中。
- **SC-004**: 100% 的保留摘要字段都能映射到当前运行时或 UI 的实际消费场景；不存在仅靠重复投影存活的摘要层。
- **SC-005**: 在本次审计范围内，subrun 领域不再同时维护两套表达同一语义的核心状态、描述符或接受结果模型。
- **SC-006**: 若 `TokenExceeded` 仍为独立业务语义，则 100% 的 core/runtime/protocol/frontend 状态表达都使用同一正式终态，不再出现内部保留、外部折叠的双轨映射。
- **SC-007**: 在本次审计范围内，subrun 公开模型不再同时暴露核心必填字段与其重复 descriptor 壳（SC-005 在 descriptor/parent_turn_id 上的具体度量），也不再通过 optional `parent_turn_id` 维持 downgrade 公开语义。
- **SC-008**: 在本次审计范围内，新发现的同类冗余项都能按同一清理原则收口为唯一正式表达，而不是继续演化成新的兼容层。
- **SC-009**: 当前有效规格、开放项和实现说明中，对已删除能力的主线宣传引用数降为 0。
- **SC-010**: 约定的主线回归场景全部通过，包括当前会话浏览、当前子执行聚焦、当前子会话直开、当前配置读写、当前消息提交与当前活跃子执行控制。
- **SC-011**: child session open target 在 core/runtime/protocol 公开合同中只保留 1 处正式字段，不再同时存在嵌套字段与外层重复字段表达同一目标。
- **SC-012**: child/subrun 相关 protocol 状态字段的字符串公开表达数降为 0；前端不再需要通过硬编码字符串集合判断状态。
- **SC-013**: compaction 原因到 durable trigger 的映射只保留 1 处集中定义；不存在多处 `Reactive -> Auto/Manual` 的并行手写映射。
- **SC-014**: `PromptMetrics` 共享字段集合在 storage、agent 与 protocol 三层只维护 1 份共享 payload 定义，不再存在三套 100% 同构字段清单。

## Assumptions

- 本次特性以当前桌面端产品流程为唯一支持基线；没有当前消费者、没有明确拥有者的外部接口不因“可能被外部使用”而自动获得保留资格。
- 本项目不要求向后兼容；旧客户端、旧历史数据和旧内部抽象在本次清理后可以直接失效，不需要额外迁移或兼容桥接，除非它们仍支撑当前主线流程。
- 当前仍应保留的摘要能力仅限真实被消费的摘要事实，例如子 Agent 终态交接摘要与父侧通知摘要；无人消费的平行摘要投影不受保护。
- 当前文档中的 archive 材料可以作为历史记录保留，但当前生效的 spec、开放项和实现说明必须与清理后的正式支持面一致。
- 若发现某个看似 legacy 的入口实际上仍支撑主线动作，本特性会在同一次交付中完成迁移与删除，而不是长期保留双轨入口。
- 你列出的 subrun 模型与 service contract 候选项默认被视为“当前 feature 的冗余清理输入”，除非后续澄清把其中个别项排除出本次范围。
- `SubRunDescriptor`、optional `parent_turn_id` 与 descriptorless / legacy downgrade 公开读模型默认视为同一组冗余兼容层，本次按彻底 cutover 处理，而不是保留读侧兼容。
- 对尚未逐项点名但符合相同模式的冗余项，本次默认采用“删除兼容壳层、保留唯一正式表达”的处理原则，不再为减少改动量而保留额外兼容结构。
- `ChildSessionNotification.open_session_id` 与 `child_ref.open_session_id` 默认视为不可接受的重复事实源，本次按删除外层重复字段处理，而不是继续维持“双写但约定相等”。
- protocol 中把 child/subrun 状态降为字符串默认视为契约缺陷，而不是 DTO 层可接受的结构性代价；本次按补齐强类型枚举处理。
- `Reactive` compaction 语义默认仍需要保留为运行时/hook 内部原因，但不应额外扩展 durable trigger 集合；落盘时按显式归一映射处理。
- `PromptMetrics` 三层字段重复默认视为应当抽取共享 payload 的设计问题，而不是 protocol 与 domain 分层不可避免的镜像成本。
