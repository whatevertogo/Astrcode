# Feature Specification: 删除死代码与兼容层收口

**Feature Branch**: `006-prune-dead-code`  
**Created**: 2026-04-10  
**Status**: Draft  
**Input**: User description: "重新定义删除死代码、无意义代码和向后兼容层的清理需求，覆盖前端路由/API、后端未消费端点、legacy 子智能体兼容路径和相关文档/测试收口。"

## Scope Overview

本特性不是一次泛泛的“代码整理”，而是一次明确的收口动作：

1. 删除当前产品路径没有消费、只有测试或文档还在自我引用的前端读模型、状态投影和后端接口。
2. 删除已经失去主线地位、仅为旧子智能体/旧子会话/旧 sub-run 语义保留的向后兼容层。
3. 保留当前仍有真实消费者的核心摘要语义，例如子 Agent 终态交接摘要与父侧通知摘要；删除没有 UI 或运行时消费者的重复摘要投影。
4. 同步收口文档、测试、夹具和术语，避免仓库继续暗示“不再支持但好像还能用”的表面能力。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 收口正式支持面 (Priority: P1)

作为维护者，我希望仓库里只保留当前产品确实在使用、并且有人负责的前后端表面能力，这样我在阅读代码和推进后续重构时，不会被“看起来存在、实际上没人用”的接口、投影和状态模型误导。

**Why this priority**: 如果正式支持面不先收口，后续每一次重构、排障和规格更新都会继续背着伪能力和伪边界前进，复杂度会反复回弹。

**Independent Test**: 对前端导航/读模型、后端接口和摘要语义做一次消费者审计；只实现这一项，也能立即让当前代码库的“正式支持范围”变得清晰、可验证。

**Acceptance Scenarios**:

1. **Given** 某个前端 API、投影函数或导航辅助结构没有任何当前 UI 或交互流程消费，**When** 完成本次清理，**Then** 该能力连同其配套引用一起被删除，而不是继续作为“以后可能会用”的预实现保留。
2. **Given** 某个后端接口没有当前产品入口、没有活跃消费者，也没有被确认仍是正式支持的外部契约，**When** 完成本次清理，**Then** 该接口被移出正式支持面，而不是继续作为空壳或未收口接口留在代码库中。
3. **Given** 同一语义存在多个摘要表达层，**When** 完成本次清理，**Then** 只有当前 UI 或运行时真正消费的摘要事实被保留，重复投影或无人消费的摘要层被删除。

---

### User Story 2 - 删除旧模型兼容层 (Priority: P1)

作为维护者，我希望系统不再为了旧子智能体模型、旧会话写入方式或旧控制路径保留双轨逻辑，这样运行时、协议和前端读模型都只围绕当前主线能力演进，而不是继续对旧世界做显式照顾。

**Why this priority**: 向后兼容层会把已经结束的迁移期永久固化成维护成本，尤其会污染运行时边界、状态类型、错误分支和测试夹具。

**Independent Test**: 只做这一项，也能让系统从“主线能力 + 兼容层”的双轨结构收束为“一个主线 + 明确失败边界”的单轨结构。

**Acceptance Scenarios**:

1. **Given** 某条读写、状态查询、取消或导航路径仅用于兼容旧子智能体/旧 sub-run/旧共享历史语义，**When** 完成本次清理，**Then** 该兼容路径被删除，而不是继续以 legacy、downgrade 或 fallback 的形式长期保留。
2. **Given** 某条旧路径仍然是当前活跃用户流程的唯一入口，**When** 进行清理，**Then** 系统先把该活跃流程迁移到唯一的正式路径，再删除旧路径，而不是直接留下功能缺口。
3. **Given** 遇到历史遗留数据或旧调用方式，**When** 这些输入落入当前主线流程，**Then** 系统宁可明确失败，也不能继续伪造 lineage、摘要、导航关系或“部分可用”的兼容视图。

---

### User Story 3 - 文档、测试与术语同步收口 (Priority: P2)

作为维护者，我希望当前文档、测试和夹具只描述仍被支持的能力与边界，这样后来者不会再根据过时 spec、骨架路由测试或 legacy 术语，把已经决定删除的东西重新扩回来。

**Why this priority**: 如果代码删了但 spec、开放项、测试夹具和术语没删，仓库会继续对人和自动化工具发出错误信号，等于把死代码以另一种形式留下来。

**Independent Test**: 只做这一项，也能让当前文档与测试基线重新对齐主线架构，为后续计划、实现和 review 提供稳定边界。

**Acceptance Scenarios**:

1. **Given** 当前文档把无人消费或已决定删除的表面能力描述成 stable、experimental 或待收口接口，**When** 完成本次清理，**Then** 当前文档不再继续宣传这些能力。
2. **Given** 某些测试、夹具或断言只是在证明已删除路径“还存在”，**When** 完成本次清理，**Then** 这些测试和夹具被删除或改写为围绕保留能力的验证。
3. **Given** 清理后的仓库仍保留若干公共表面能力，**When** 维护者查看当前规格，**Then** 每一项保留能力都能找到清晰用途、消费者类型和所属边界，而不是靠历史上下文猜测其存在理由。

### Edge Cases

- 某个接口当前没有前端消费者，但确实仍承担正式的外部契约职责；如果不能明确其拥有者、用途和存续理由，就应按死代码处理。
- 某个 legacy 路径看起来只是兼容层，但当前活跃流程仍通过它完成关键动作；此时必须先迁移当前流程，再删除旧路径。
- 同一个“summary”字段在不同层次出现时，必须区分“当前仍被消费的摘要事实”和“仅剩重复投影的摘要外壳”，避免误删核心能力。
- 旧数据、旧客户端或旧测试在兼容层删除后可能直接失败；系统可以不保证继续可用，但不能继续伪造关系、拼接假摘要或输出误导性的部分结果。
- 当前文档可以保留归档材料作为历史记录，但当前有效规格、开放项和实现说明不能再把已删除能力写成仍在维护的现状。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 先产出一份清理清单，把候选表面能力划分为“立即删除”“先迁移后删除”“明确保留”三类，并以当前真实消费者与责任边界作为判定依据。
- **FR-002**: 任何没有当前产品消费者、没有明确拥有者、也没有被确认仍属正式支持契约的前端或后端表面能力 MUST 在本次清理中删除。
- **FR-003**: 仅为未来可能性预埋的空壳接口、骨架路由、占位投影和未落地控制面 MUST 删除，而不是继续以未启用、实验性或待后续实现的形式保留。
- **FR-004**: 当前仍被运行时或 UI 消费的核心摘要事实 MUST 保留；没有消费者的重复摘要投影、重复摘要 API 或平行摘要读模型 MUST 删除。
- **FR-005**: 前端导航相关代码 MUST 只保留当前状态驱动导航实际需要的状态、过滤与投影；没有真实消费的路由化抽象、辅助读模型或子视图投影 MUST 删除。
- **FR-006**: 后端对外暴露的接口 MUST 只保留当前正式支持的能力；无人消费、无人认领或只剩测试自证的接口 MUST 从正式支持面移除。
- **FR-007**: 已删除的表面能力 MUST 不再仅以类型、注释、文档、测试、夹具或错误码的形式残留在主线代码中。
- **FR-008**: 仅为旧子智能体模型、旧共享历史语义、旧状态降级语义或旧控制路径服务的向后兼容层 MUST 从运行时、协议、服务端和前端读模型中同步移除。
- **FR-009**: 对已经决定不再支持的旧输入，系统 MUST 停止提供伪完整的兼容展示；若这些输入进入主线流程，系统 MUST 以明确失败替代猜测性恢复、猜测性 lineage 或猜测性摘要。
- **FR-010**: 若某个 legacy 风格的接口仍是当前活跃流程的唯一入口，系统 MUST 在同一次特性中提供唯一的正式替代路径，并将活跃流程切换完成后再删除旧入口。
- **FR-011**: 清理完成后，每个仍保留的关键用户动作 MUST 只有一条正式路径，不允许同时存在“主线入口”和“兼容入口”并行维护。
- **FR-012**: 当前有效文档 MUST 与清理结果一致，不得继续把已删除、无人消费或不再维护的能力描述为 stable、experimental、planned 或可选入口。
- **FR-013**: 当前测试基线 MUST 围绕保留能力和明确失败边界重建，不得继续为了维持已删除路径的存在感而保留专门测试。
- **FR-014**: 每个被明确保留的公共表面能力 MUST 在当前规格中说明其用途、消费者类型和所属边界，避免后来者只能通过历史提交推断其存在理由。
- **FR-015**: 本次清理 MUST 覆盖前端状态导航面、前端会话/子会话 API 面、服务端 HTTP 面、运行时兼容分支、协议暴露面以及与之对应的当前文档与测试资产。
- **FR-016**: 本次清理 MUST 不破坏当前仍在使用的主线能力，包括当前会话浏览、当前子执行浏览、当前子会话直开、当前配置读写、当前消息提交流程，以及当前活跃的子执行控制动作。

### Key Entities *(include if feature involves data)*

- **正式支持面**: 当前产品或运行时真正依赖、且有明确拥有者与用途说明的前端状态、接口、协议或交互入口。
- **死代码表面能力**: 没有真实消费者，只剩类型、注释、测试、文档或占位实现证明其存在的能力。
- **先迁移后删除能力**: 设计上已经过时，但当前活跃流程仍暂时依赖，因此必须先切换到唯一替代路径再删除的能力。
- **兼容层工件**: 为旧子智能体模型、旧会话写入语义、旧状态降级展示或旧控制路径保留的额外类型、分支、接口、错误码或夹具。
- **摘要事实**: 当前仍被消费、用于表达子 Agent 终态交接或父侧通知的信息。
- **重复摘要投影**: 与现有摘要事实表达同一语义，但当前没有 UI 或运行时消费者的额外投影层。
- **清理清单**: 对所有候选表面能力给出的保留、迁移或删除决策集合，是实施和验证本特性的审计基线。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 在本次审计范围内，100% 的保留表面能力都能指向当前真实消费者与明确边界；不存在“没人用但先留着”的保留项。
- **SC-002**: 在本次审计范围内，前端导出 API/投影与后端公开接口中，0 个能力只剩测试或文档引用而没有当前产品消费者。
- **SC-003**: 在本次审计范围内，0 个仅为旧子智能体/旧共享历史/旧 sub-run 语义保留的兼容入口继续留在主线能力面中。
- **SC-004**: 100% 的保留摘要字段都能映射到当前运行时或 UI 的实际消费场景；不存在仅靠重复投影存活的摘要层。
- **SC-005**: 当前有效规格、开放项和实现说明中，对已删除能力的主线宣传引用数降为 0。
- **SC-006**: 约定的主线回归场景全部通过，包括当前会话浏览、当前子执行聚焦、当前子会话直开、当前配置读写、当前消息提交与当前活跃子执行控制。

## Assumptions

- 本次特性以当前桌面端产品流程为唯一支持基线；没有当前消费者、没有明确拥有者的外部接口不因“可能被外部使用”而自动获得保留资格。
- 本项目不要求向后兼容；旧客户端、旧历史数据和旧内部抽象在本次清理后可以直接失效，不需要额外迁移或兼容桥接，除非它们仍支撑当前主线流程。
- 当前仍应保留的摘要能力仅限真实被消费的摘要事实，例如子 Agent 终态交接摘要与父侧通知摘要；无人消费的平行摘要投影不受保护。
- 当前文档中的 archive 材料可以作为历史记录保留，但当前生效的 spec、开放项和实现说明必须与清理后的正式支持面一致。
- 若发现某个看似 legacy 的入口实际上仍支撑主线动作，本特性会在同一次交付中完成迁移与删除，而不是长期保留双轨入口。
