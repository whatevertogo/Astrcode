# Feature Specification: 子 Agent 独立子会话模型

**Feature Branch**: `003-subagent-child-sessions`  
**Created**: 2026-04-08  
**Status**: Draft  
**Input**: User description: "Refactor subagent architecture so child agents run as durable child sessions with independent transcripts, tool-based communication, wait/resume/close control, parent-child handoff summaries, and a UI model that shows thinking, tool activity, and final replies while preparing the runtime model for future fork agents."

## User Scenarios & Testing

### User Story 1 - 稳定委派与交付 (Priority: P1)

作为主 agent 的使用者和维护者，我希望子 agent 在被创建后成为独立的子会话，即使父 turn 已经结束，子 agent 仍能继续工作，并在完成时向主 agent 交付稳定、可追溯的最终回复。

**Why this priority**: 这是当前问题的根因修复点。只有先把“子 agent 不是父 turn 附属执行流”建立起来，后续的恢复、查看、打回和 UI 改善才有可靠基础。

**Independent Test**: 创建一个会在父 turn 结束后继续运行的子 agent；验证其不会被自动取消，并且在结束后主 agent 能收到一次且仅一次终态交付。

**Acceptance Scenarios**:

1. **Given** 主 agent 启动了一个长时间运行的子 agent，**When** 父 turn 正常结束，**Then** 子 agent 继续保持运行状态，直到显式取消或进入终态。
2. **Given** 子 agent 已完成自己的工作，**When** 系统生成交付结果，**Then** 主 agent 收到包含子 agent 最终回复、状态和可查看引用的结构化交付，而不是零散事件片段。
3. **Given** 子 agent 失败或被终止，**When** 主 agent 收到终态通知，**Then** 通知中包含失败原因或最后有效进展，避免出现“只看到探索看不到结论”的情况。
4. **Given** 父 turn 已结束但父 agent 仍需继续服务用户，**When** 子 agent 产生新的交付消息，**Then** 系统重新激活父 agent 的工作流，使其基于该交付继续决策、回复和推进用户任务。
5. **Given** 主 agent 需要启动一个新的子 agent，**When** 它通过统一的子 agent 入口发起委派，**Then** 系统以明确的启动契约记录本次委派的任务说明、目标 profile、执行模式和子会话引用，而不是把子 agent 当作父会话中的隐式副作用。
6. **Given** 主 agent 需要与某个子 agent 协作，**When** 它执行协作操作，**Then** 这些协作在契约层统一表现为 tool 调用，而 runtime 在内部用定向投递机制完成实际送达与唤醒。

---

### User Story 2 - 可查看的子会话视图 (Priority: P1)

作为维护者，我希望每个子 agent 都有独立、可重放、可打开的子会话视图，这样我可以从父会话进入子会话，查看它的思考摘要、工具活动和最终答复，而不是阅读原始协议 JSON。

**Why this priority**: 即使生命周期修复了，如果子 agent 的内容仍然只能混在父流中显示，维护者仍然难以理解系统行为，也难以定位问题。

**Independent Test**: 运行至少一个成功的子 agent 和一个失败的子 agent；从父会话进入各自的子会话视图，验证都能看到清晰的时间线和最终状态。

**Acceptance Scenarios**:

1. **Given** 父会话中存在多个子 agent，**When** 用户选择其中一个子 agent，**Then** 系统打开对应的子会话视图，并显示该子 agent 的思考摘要、工具活动和最终答复。
2. **Given** 子 agent 已经结束且父会话被重新加载，**When** 用户再次打开该子 agent，**Then** 仍然能看到完整的子会话历史和终态信息。
3. **Given** 默认会话视图，**When** 子 agent 结果在父会话中呈现，**Then** 只展示结构化摘要和交付信息，原始协议 JSON 默认隐藏。
4. **Given** 父会话中存在多个子 agent 摘要项，**When** 用户展开其中一个子 agent，**Then** 系统只展开该目标子 agent 的详细时间线，不影响其他子 agent 的摘要展示状态。
5. **Given** 用户已经查看过某个子 agent 的详细内容，**When** 父会话刷新、重载或稍后恢复，**Then** 该子 agent 的身份、终态和可进入的子会话入口保持稳定，不会因为 UI 状态变化而丢失真实会话内容。

### User Story 3 - 主子双向协作 (Priority: P2)

作为主 agent，我希望能继续向一个正在运行、已完成但可恢复、或需要返工的子 agent 发送补充要求，让它继续完善结果，而不是每次都重新创建新的子 agent。

**Why this priority**: 没有持续通信能力，子 agent 只能“一次性跑完”，无法形成真正的协作回路，也无法实现你讨论中的“打回去重新完善”。

**Independent Test**: 创建一个子 agent，先让它完成初版答复，再由主 agent 向其发送修订要求，验证同一个子 agent 会话继续产出新的终态交付。

**Acceptance Scenarios**:

1. **Given** 子 agent 正在运行，**When** 主 agent 发送补充说明或修订要求，**Then** 子 agent 接收到新的输入并继续在原子会话中工作。
2. **Given** 子 agent 已经完成但未被关闭，**When** 主 agent 要求其补充证据或改写答案，**Then** 系统恢复同一个子 agent 会话，而不是强制创建新实例。
3. **Given** 主 agent 需要等待某个子 agent 的结果，**When** 调用等待能力，**Then** 系统只等待目标子 agent，不影响其他子 agent 或父会话的正常工作。
4. **Given** 主 agent 决定停止某个子 agent，**When** 调用关闭或取消能力，**Then** 只有目标子 agent 进入终态，不会误伤同批次的其他子 agent。
5. **Given** 主 agent 已收到某个子 agent 的交付，**When** 该子 agent 对当前任务已不再有帮助，**Then** 主 agent 默认主动关闭该子 agent；若后续仍可能继续利用其上下文，则保持其可恢复状态而不关闭。

### User Story 4 - 层级协作与级联关闭 (Priority: P2)

作为架构维护者，我希望父 agent 和子 agent 之间存在受控的双向协作能力，并且关闭语义按 agent 层级树传播，而不是按 turn 边界传播，这样多级 agent 链路才能稳定运行和正确收口。

**Why this priority**: 只解决“父发给子”还不够，多级 agent 体系还需要“子如何把结果上交给父”以及“父关闭时子孙如何退出”的统一规则，否则未来会再次出现 turn 误绑定、悬挂实例和关闭顺序混乱的问题。

**Independent Test**: 构造一条至少三层的 agent 链路，验证下级 agent 能通过受控入口把交付回传给上级 agent；随后关闭上级 agent，验证系统按层级自底向上完成级联关闭。

**Acceptance Scenarios**:

1. **Given** 子 agent 需要将阶段性结果或最终结果返回给父 agent，**When** 它使用上行协作入口，**Then** 父 agent 收到统一语义的上行消息，而不是依赖读取零散底层事件。
2. **Given** 存在父 agent、子 agent 和孙 agent 的层级关系，**When** 父 agent 被显式关闭，**Then** 系统先关闭最底层后代，再逐级关闭上层子 agent，直到该父 agent 的整棵子树全部进入终态。
3. **Given** 某个中间层 agent 被关闭，**When** 它仍有后代在运行，**Then** 只有该 agent 及其后代链路被级联关闭，平级或其他分支的 agent 不受影响。
4. **Given** 某个子 agent 已向父 agent 完成交付，**When** 父 agent 决定继续保留它，**Then** 该子 agent 仍可参与后续协作，而不会因一次交付被自动关闭。
5. **Given** 多个下级 agent 同时向上级回传消息，**When** 某个下级 agent 产生新的交付或续做请求，**Then** 只有它的直接父 agent 会收到并消费该消息，其他平级 agent 或祖先链路不会误消费该输入。

---

### User Story 5 - 为未来 Fork Agent 复用同一底座 (Priority: P3)

作为架构维护者，我希望未来的 fork agent 不是另一套并行系统，而是复用同一个子会话、状态机和交付模型，只在“创建来源”上与普通子 agent 不同。

**Why this priority**: 如果现在不把 lineage 和创建模式纳入统一模型，后续加入 fork agent 时很可能再次引入第三套生命周期和展示语义。

**Independent Test**: 审查子会话模型后，能明确区分“新任务创建的子会话”和“继承父上下文创建的子会话”，且二者共享相同的查看、等待、恢复、关闭和交付规则。

**Acceptance Scenarios**:

1. **Given** 一个普通子 agent 会话和一个继承上下文的子会话，**When** 系统记录它们的来源信息，**Then** 两者都能通过同一种生命周期和查看规则被管理。
2. **Given** 未来新增 fork 创建入口，**When** 维护者接入该入口，**Then** 无需重新定义子会话展示、通信和终态交付协议。

### Edge Cases

- 父 turn 结束、父会话刷新或父 UI 关闭时，仍在运行的子 agent 不应被隐式取消。
- 子 agent 在没有生成最终文本前失败、超时或被关闭时，系统仍需提供终态原因和最后有效进展。
- 多个子 agent 由同一个父 turn 同时启动时，等待、关闭、恢复和通知必须按目标子 agent 精确作用。
- 主 agent 向已处于终态的子 agent 发送新要求时，系统需要明确恢复原会话，而不是悄悄丢弃消息或新建重复实例。
- 子 agent 的最后一个有效输出如果不是完整答复，系统需要选择最后可读的交付文本，避免父会话只收到空结果。
- 会话回放、重载或崩溃恢复时，子 agent 不能丢失自己的终态、回复内容或父子关系。
- 子 agent 在父 turn 已结束后回传新结果时，系统需要确保父 agent 被重新激活，而不是让结果停留在无人消费的通知队列中。
- 主 agent 收到子 agent 交付后，如果不再需要该子 agent，上层行为需要有明确的默认关闭策略，避免无意义的子 agent 长期悬挂。
- 存在多级 agent 链路时，关闭传播必须沿 agent 所有权树进行，而不是沿 turn 生命周期进行。
- 子 agent 向父 agent 的上行消息必须通过统一、受控的协作入口进入父 agent，避免下级直接操纵上级内部执行状态。
- 多个 agent 同时存在待处理协作输入时，系统必须按目标 agent 精确路由和消费消息，避免出现父子串话、平级误收或同一交付被重复消费。
- 父视图中的折叠、展开和摘要呈现状态不得影响子会话真实内容的持久化；即使 UI 不展示全部细节，子会话原始可读时间线仍需完整保留并可重新进入。
- 子 agent 的审批、工具边界、工作目录或隔离策略可以与父 agent 不同，但这些策略必须属于子会话自己的执行边界，而不是隐式继承且无法追踪。
- 主会话不应持续吞入子会话的中间细节流；父侧只接收可消费的任务通知、摘要和终态结果，子会话细节保留在自己的完整时间线中。
- 即使 agent 间协作在模型契约上表现为 tool，runtime 也不应把内部事件传输重新包装成一整套递归 tool 执行流；否则会增加延迟、噪音和恢复复杂度。

## Requirements

### Functional Requirements

- **FR-001**: 系统 MUST 在每次创建子 agent 时生成独立的子会话标识，该标识独立于父 turn 和父工具调用。
- **FR-002**: 系统 MUST 将子 agent 的状态、历史、最终答复和关联产物作为独立子会话持久化，而不是仅作为父会话中的临时事件流保存。
- **FR-003**: 父 turn 的正常完成 MUST NOT 自动取消仍在运行的子 agent；只有显式关闭、取消、会话销毁策略或子 agent 自身终态才能结束其生命周期。
- **FR-004**: 系统 MUST 为每个子 agent 保存父会话引用、来源 turn、创建来源类型和当前状态，以支持后续查看、恢复和调试。
- **FR-005**: 主 agent MUST 能通过工具能力创建子 agent，并获得一个可用于后续通信、等待、恢复和关闭的稳定引用。
- **FR-006**: 主 agent MUST 能向目标子 agent 发送后续输入，以补充要求、要求返工或继续同一任务。
- **FR-007**: 系统 MUST 支持针对指定子 agent 的等待能力，并在目标子 agent 进入终态后返回其终态状态和最新交付摘要。
- **FR-008**: 系统 MUST 支持针对指定子 agent 的关闭或取消能力，且该操作不得影响未被选中的其他子 agent。
- **FR-009**: 已完成但未被关闭的子 agent MUST 保持可恢复状态，允许主 agent 在原会话内继续追加任务或修订要求。
- **FR-010**: 当子 agent 进入终态时，系统 MUST 向父会话生成结构化交付摘要，至少包含子 agent 身份、终态状态、最终答复或失败原因、以及进入子会话的查看入口。
- **FR-011**: 默认父会话视图 MUST 以摘要形式展示子 agent 的思考、工具活动和最终答复，MUST NOT 默认展示原始协议 JSON。
- **FR-012**: 独立子会话视图 MUST 展示子 agent 的思考摘要、工具活动、最终答复和终态信息，并支持在会话重载后重新打开。
- **FR-013**: 系统 MUST 在子 agent 无法产出完整最终答复时，尽可能保留最后有效的可读结果并连同失败原因一起交付给父会话。
- **FR-014**: 会话重放与恢复 MUST 重建子 agent 的可查看状态、终态状态和父子关系，避免出现终态丢失、重复通知或回复缺失。
- **FR-015**: 系统 MUST 区分“新任务创建的子会话”和“继承上下文创建的子会话”两类来源，并让两类来源共享同一生命周期、通信和查看模型。
- **FR-016**: 系统 MUST 将主会话与子会话之间的通信顺序持久化，确保补充要求、子 agent 响应和终态通知在回放时顺序一致。
- **FR-017**: 当子 agent 向父会话回传新的交付消息时，系统 MUST 在需要时重新激活父 agent 的执行流，使父 agent 能基于新交付继续工作并服务用户。
- **FR-018**: 系统 MUST 为父 agent 提供统一的子 agent 交付入口，使子 agent 的回复以一致的父侧输入语义进入父 agent，而不是作为仅供 UI 展示的孤立通知存在。
- **FR-019**: 父 agent 在收到子 agent 交付后 MUST 能显式决定“继续保留”或“主动关闭”该子 agent；当父 agent 不再需要该子 agent 时，默认行为应为主动关闭以避免悬挂实例。
- **FR-020**: 如果父 agent 仍可能继续利用某个子 agent 的上下文或后续工作成果，系统 MUST 允许父 agent 保持该子 agent 为可恢复状态而不强制关闭。
- **FR-021**: 系统 MUST 提供受控的上行协作入口，使子 agent 能将阶段性结果、最终结果或继续请求发送给直接父 agent。
- **FR-022**: 子 agent 发送给父 agent 的上行协作消息 MUST 进入父 agent 的统一输入语义，而不是绕过父 agent 的正常决策流程直接修改其内部状态。
- **FR-023**: 系统 MUST 将 agent 之间的父子所有权关系持久化为层级树，以支持多级通信、恢复和关闭传播。
- **FR-024**: 当某个 agent 被显式关闭时，系统 MUST 按其后代层级执行级联关闭，并以叶子优先的顺序完成收口。
- **FR-025**: 级联关闭 MUST 以 agent 层级所有权为依据，MUST NOT 以父 turn 是否结束作为关闭判定条件。
- **FR-026**: 当某个中间层 agent 被关闭时，系统 MUST 仅关闭该 agent 及其后代子树，不得影响其他平级或无关分支。
- **FR-027**: 父 agent 在收到下级 agent 的上行交付后 MUST 能继续通过下行协作能力与该下级 agent 对话，除非父 agent 已显式关闭它。
- **FR-028**: 系统 MUST 为每个 agent 维护按目标身份路由的协作输入语义，确保某条上行或下行消息只会投递给被明确指定的目标 agent。
- **FR-029**: 某个 agent 接收到的协作输入 MUST 只被该 agent 自身消费一次；系统 MUST 防止同一交付被父 agent 重复消费，或被平级 agent、祖先 agent 误消费。
- **FR-030**: 父会话中的子 agent 展示 MUST 采用“父侧摘要投影 + 子侧完整时间线”的双层视图模型；父侧默认只显示摘要、思考要点、工具活动概览和最终答复，详细内容仅在目标子会话中展开。
- **FR-031**: 子会话的身份、终态、最终答复、可读时间线和查看入口 MUST 独立于父视图的折叠或展开状态持久化，确保重载、恢复或跨视图跳转后仍可稳定打开同一子会话。
- **FR-032**: 系统 MUST 支持为运行中、已完成但可恢复、以及已终止的子 agent 生成一致的父侧摘要投影，使维护者无需阅读原始协议事件即可区分其当前状态、最近进展和最终结论。
- **FR-033**: 当子 agent 被恢复继续工作时，系统 MUST 复用原有子会话身份和历史，而不是生成新的并列子会话；父视图中的查看入口也 MUST 持续指向同一子会话。
- **FR-034**: 系统 MUST 将“启动子 agent”定义为明确的协作入口，而不是隐式副作用；每次启动都必须生成可追踪的启动契约，至少包含任务说明、目标 agent 类型或 profile、执行模式以及生成的子会话引用。
- **FR-035**: 子 agent 启动契约 MUST 支持表达子会话自己的执行边界，包括但不限于运行前台或后台、工具边界、审批策略、工作目录或隔离范围；这些边界必须归属于子会话自身并可在恢复时重建。
- **FR-036**: 主会话 MUST 只接收对子 agent 可消费的任务通知、摘要更新和终态交付；系统 MUST NOT 将子会话的完整中间事件流直接混入父会话主消息流。
- **FR-037**: 子会话持久化 MUST 同时保留独立可读历史和恢复所需的运行元数据，使系统在重载后既能展示子会话完整时间线，也能重建其执行状态、边界和输出位置。
- **FR-038**: 当系统支持 fork 形式的子会话创建时，fork MUST 复用同一子会话生命周期和交付模型，并额外记录其来源快照或 lineage 信息，而不是形成独立的第三套执行体系。
- **FR-039**: agent 之间的启动、补充要求、等待、关闭和上行交付 MUST 在模型契约层统一表现为明确的 tool 能力，以保持 prompt surface、权限治理和审计语义一致。
- **FR-040**: 运行时 MUST 使用面向目标 agent 的 inbox、mailbox 或等价定向投递机制来实现这些 tool 协作语义，而不是要求父子 agent 之间通过暴露完整内部事件流来互相通信。
- **FR-041**: tool 契约层与 runtime 投递层 MUST 明确分离；前者负责定义模型可调用的协作能力与结果语义，后者负责消息送达、唤醒、恢复和去重，不得混为同一抽象。
- **FR-042**: 父会话接收到的子 agent 结果 MUST 表现为可消费的 tool 结果语义或任务通知投影，而不是子会话内部 progress 事件、原始 JSON 负载或递归展开的内部工具流。
- **FR-043**: 当协作工具触发对子 agent 或父 agent 的投递时，系统 MUST 保证投递结果具备幂等与去重语义，避免因恢复、重试或重复通知而让同一协作操作被多次消费。

### Key Entities

- **父会话**: 发起子 agent、接收交付摘要、并保留子会话引用的主协作上下文。
- **子会话**: 承载单个子 agent 独立生命周期、历史、状态和最终答复的协作实体。
- **子 agent 引用**: 主 agent 用于后续发送输入、等待、恢复或关闭目标子会话的稳定标识。
- **交付摘要**: 子 agent 进入终态后回传给父会话的结构化结果，包含状态、最终答复或失败原因，以及子会话查看入口。
- **通信记录**: 主会话与子会话之间追加输入、接收结果和终态通知的顺序化交互记录。
- **创建来源**: 记录子会话是基于全新任务创建还是基于继承上下文创建的 lineage 元信息，用于支持未来 fork agent。
- **Agent 层级树**: 描述 agent 之间直接父子所有权关系的层级结构，用于驱动多级通信、恢复和级联关闭。
- **上行协作消息**: 由下级 agent 通过受控入口发送给直接父 agent 的结构化输入，用于交付结果、请求续做或发起进一步协作。
- **Agent 定向输入**: 绑定到某个目标 agent 身份的协作输入单元，用于保证父子通信、恢复通知和等待结果都按目标 agent 精确路由。
- **父侧摘要投影**: 子会话在父会话中的简化展示形态，只呈现维护者继续决策所需的思考摘要、工具活动概览、终态和查看入口。
- **子侧完整时间线**: 子会话自身保留的完整可读历史，用于展开查看子 agent 的详细推理、工具调用过程和最终答复。
- **子 agent 启动契约**: 主 agent 发起委派时生成的结构化启动描述，记录任务说明、目标 profile、执行模式和所创建的子会话引用。
- **子会话执行边界**: 绑定到某个子会话的运行策略集合，包括审批、工具范围、工作目录、隔离方式和前后台模式，用于保证子会话恢复与治理时语义一致。
- **父侧任务通知**: 由子会话向父会话投影的可消费事件，用于传达摘要更新、完成、失败、中止或续做信号，而不是替代子会话完整历史。
- **协作工具契约层**: 面向模型暴露的 agent 协作能力集合，定义 spawn、send、wait、close、deliver 等可调用入口及其结果语义。
- **运行时投递层**: 在 runtime 内部负责 agent 定向投递、唤醒、恢复、去重和收件箱治理的实现层，不直接暴露为父会话原始消息流。

## Success Criteria

### Measurable Outcomes

- **SC-001**: 在覆盖“父 turn 先结束、子 agent 后完成”的验证场景中，100% 的子 agent 能继续运行至自身终态，不会因父 turn 完成而被提前结束。
- **SC-002**: 在成功、失败、取消三类终态场景中，100% 的子 agent 都会向父会话生成一次且仅一次结构化交付摘要。
- **SC-003**: 在包含至少 10 次子 agent 创建、补充要求、等待、恢复和关闭的验证流程中，目标操作的命中率达到 100%，不存在误操作到其他子 agent 的情况。
- **SC-004**: 在会话重载或恢复后的验证场景中，100% 的已存在子会话都能重新打开并显示其最终答复或失败原因。
- **SC-005**: 默认父会话和子会话界面在所有验证场景下都隐藏原始协议 JSON，仅展示结构化的思考摘要、工具活动和最终答复。
- **SC-006**: 在“子 agent 初次答复后被主 agent 打回完善”的验证场景中，95% 以上的案例能在同一子会话内完成二次交付，而无需创建新的重复子 agent。
- **SC-007**: 在“父 turn 已结束、子 agent 后续回传结果”的验证场景中，父 agent 对新交付的恢复处理成功率达到 100%，不会出现结果被接收但父 agent 未继续工作的情况。
- **SC-008**: 在包含多个子 agent 的验证流程中，父 agent 对“无用则关闭、有用则保留”的收口决策可被稳定重放，且不会产生长期无主的悬挂子 agent。
- **SC-009**: 在三层以上 agent 链路的验证场景中，100% 的下级 agent 都能通过统一的上行协作入口把消息送达其直接父 agent。
- **SC-010**: 在包含父 agent、子 agent、孙 agent 的级联关闭验证中，100% 的关闭流程都按叶子优先顺序完成，且不会误关闭无关分支。
- **SC-011**: 在至少包含 3 个并发子 agent 的验证场景中，100% 的上行交付和下行补充要求都只会被目标 agent 消费一次，不会出现串话、误投递或重复消费。
- **SC-012**: 在父视图折叠显示、展开查看、重载恢复三类验证场景中，100% 的子 agent 都能维持同一个子会话入口与身份，并且父视图无需展示原始协议 JSON 也能让维护者识别状态、最近进展和最终回复。
- **SC-013**: 在包含成功、失败和后台运行三类子 agent 的验证场景中，100% 的父会话只接收结构化任务通知、摘要更新和终态交付，不会混入子会话的完整中间事件流。
- **SC-014**: 在子 agent 使用独立审批、工具边界或工作区隔离的验证场景中，100% 的子会话都能在恢复后重建相同执行边界，且不会错误退化为父会话默认策略。
- **SC-015**: 在 spawn、send、wait、close、deliver 五类协作场景中，100% 的模型侧交互都通过统一的 tool 契约完成，不依赖父会话直接读取子会话内部事件流。
- **SC-016**: 在恢复、重试和并发通知场景中，100% 的协作投递都能保持一次有效消费语义，不会因为 runtime 重放而让父子 agent 重复执行同一协作操作。

## Clarifications

### Session 2026-04-08

- Q: 子 agent 在父 turn 结束后回传结果时，父侧应如何处理？ → A: 子 agent 的交付必须以统一的父侧输入语义重新送达父 agent，在需要时重新激活父 agent 的执行流，让父 agent 继续判断下一步并服务用户，而不是只留下静态通知。
- Q: 主 agent 收到子 agent 的交付后，默认应如何管理该子 agent？ → A: 主 agent 应默认承担收口责任；如果判断该子 agent 已无继续价值，应主动关闭；如果后续仍可能继续利用其上下文或工作成果，则保持可恢复状态而不关闭。
- Q: 父子 agent 之间是否需要双向对话能力？ → A: 需要，但应采用受控协作入口：父 agent 通过下行能力与子 agent 对话，子 agent 通过上行协作入口向直接父 agent 交付结果或请求续做，避免下级直接操纵上级内部状态。
- Q: 关闭传播应如何定义？ → A: 关闭传播应基于 agent 层级所有权树而不是 turn；当某个 agent 被显式关闭时，应从最底层后代开始逐级收口，只影响该 agent 及其后代子树。
- Q: 父子 agent 的消息路由和视图投影应采用什么原则？ → A: 协作消息应按目标 agent 身份精确投递并只消费一次；父会话只保留便于继续决策的摘要投影，完整可读历史保留在对应子会话中，通过稳定入口展开查看。
- Q: 从其他 CLI 设计中，最值得借鉴到本特性的 agent 边界是什么？ → A: 子 agent 应被建模为通过明确协作入口启动的独立子会话，拥有自己的 transcript、任务状态和执行边界；父会话只接收可消费的任务通知、摘要和终态结果，而不是直接吞入子会话细节流。
- Q: agent 间协作是否应统一通过 tool 暴露？ → A: 应当统一通过 tool 暴露协作契约，但 runtime 内部应使用定向 inbox 或等价投递层来实现送达、唤醒与恢复；父会话只消费投影后的 tool 结果或任务通知，而不直接承载子会话原始内部事件流。

## Assumptions

- 本特性的范围是建立统一的子会话生命周期、持久化和交互模型；完整的 fork agent 用户入口不在本次交付范围内，但必须被该模型直接支持。
- 现有历史会话可以保留旧格式读取策略，但新创建的子 agent 一律遵循新的独立子会话模型。
- 原始协议事件或调试元数据可以继续保留用于诊断，但默认用户界面不直接展示这些内容。
- 子会话在同一父会话 lineage 下保持唯一归属，即使被多次恢复，也不会改变其父会话来源。
